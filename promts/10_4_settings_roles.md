# Этап 10.4 — Разнесение настроек администратора и оператора

> Проект **«Аудиопротокол»**, улучшения после v1
> ([`../docs/recommendations.md`](../docs/recommendations.md), п. 4). Цель —
> превратить визуальное разделение секций «Оператор»/«Администратор» на экране
> настроек в **реальное разграничение доступа**: оператор не может изменить
> инфраструктуру и безопасность станции, администратор сопровождает парк
> станций с аудитом.

## Контекст

Экран настроек (`04`) уже разделён на секции визуально (решение заказчика),
но без контроля доступа. Роль пользователя приходит из входа оператора
(этап `10.3`); для оффлайна и аварийного доступа — локальный фолбэк.
Параметры — реестр [`../docs/configuration.md`](../docs/configuration.md);
новые параметры этапа **добавляются в реестр** (кандидаты:
`admin.access` = роль/PIN, `admin.pin_hash_source`).

## Результат этапа (Deliverables)

1. **Контроль доступа к администраторским настройкам.** Источник прав — роль
   из `ex_system` (`10.3`); локальный фолбэк для оффлайна — админ-PIN/парольная
   фраза станции (хранится как хеш, задаётся при развёртывании). Для оператора
   админ-секция видна **read-only** (прозрачность без возможности сломать).
2. **Разделение экранов.** «Настройки» (оператор: устройство, справочники
   меток/ролей, удобства) и «Администрирование» (сервер/URL, шифрование,
   ретеншн, пороги надёжности, карта дорожек многоканала, разрешение экспорта
   `10.2`, политика auth `10.3`). Реестр `configuration.md` — разметить каждую
   настройку областью «оператор/админ».
3. **Журнал изменений настроек.** Каждое сохранение — событие журнала
   (`integrity.event_log`): кто, когда, какие параметры изменил
   (старое → новое значение, без секретов).
4. **Опасные изменения — с подтверждением.** Смена URL сервера, отключение
   шифрования, смягчение ретеншна — явное подтверждение + событие журнала.
5. **Экспорт/импорт профиля станции.** JSON без секретов (ключи/PIN не
   выгружаются): тиражирование конфигурации на залы, восстановление станции.
   Импорт — только администратором, с журналом.
6. **Тесты.** Оператор не может изменить админ-параметр (UI и IPC-уровень —
   защита не только кнопками); PIN-фолбэк в оффлайне; журнал изменений пишется;
   экспорт/импорт профиля идемпотентен и не содержит секретов.

## Шаги

1. Модель прав в ядре: роль сессии (`10.3`) + проверка админ-PIN; **гейт на
   IPC-командах** сохранения настроек (UI-запрет недостаточен).
2. Разметка реестра настроек областями оператор/админ (schema + доки).
3. UI: два экрана, read-only представление админ-секции, диалог разблокировки.
4. Журнал изменений + подтверждение опасных изменений.
5. Экспорт/импорт профиля станции.
6. Тесты.

## Критерии приёмки (Acceptance)

- Смена админ-параметра без прав отклоняется на уровне ядра (IPC), а не только
  скрыта в UI.
- Оператор видит текущую конфигурацию станции, но меняет только свою секцию.
- Все изменения настроек — в журнале событий с авторством; опасные — после
  явного подтверждения.
- Профиль станции переносится экспорт→импорт без секретов и даёт идентичную
  конфигурацию.
- Параметры — из реестра; `cargo clippy` чисто; тесты зелёные.

## Вне объёма этапа

- Централизованное управление конфигурацией парка станций с сервера — фаза
  «эксплуатация» (задел: профиль станции из п. 5).
- Управление ролями/учётками — `ex_system`.

## Решено

- Разграничение обязано работать и **в оффлайне** (локальный PIN-фолбэк) —
  станции живут в залах без сети.
- Секреты (ключи шифрования, PIN) в профиль станции не входят.

## Открытые вопросы

- Какая роль `ex_system` считается администратором станции (существующая
  `users/roles.py` или новая) — согласовать с `ex_system`.
- Процедура **сброса забытого PIN** (переустановка? сервисный ключ?) — от
  заказчика/эксплуатации.
