# Этап 02 — Надёжность записи («бесперебойность»)

> Проект **«Аудиопротокол»**. Цель — превратить базовый захват (`01`) в
> неубиваемую запись: конечный автомат сессии, аварийно-устойчивый журнал,
> восстановление после сбоя/питания, watchdog, обработка отказа устройства,
> контроль места, опц. дублирующая дорожка. Это **нефункциональное ядро**
> продукта — критерий «не потерять заседание».

## Контекст

Архитектура и принципы — [`../CLAUDE.md`](../CLAUDE.md) (раздел «Принципы
бесперебойности»). Все пороги/таймауты — из реестра
[`../docs/configuration.md`](../docs/configuration.md) (раздел «Запись и
надёжность»). Опирается на `01` (`audio/`, `recorder/segment_writer.rs`).
Код — `src-tauri/recorder/` + `src-tauri/reliability/`.

**Граница с `03`:** здесь — **журнал восстановления** (минимально достаточный,
чтобы возобновить сессию и знать состав сегментов). Полный манифест с хешами,
шифрование at-rest и ретеншн — этап `03` (расширяет тот же журнал).

## Результат этапа (Deliverables)

1. **Конечный автомат сессии.** Состояния `Idle → Recording ⇄ Paused →
   Stopping → Stopped`, плюс `Error` и `Recovering`. Чёткие переходы, события
   в UI (`capture_state`), запрет некорректных переходов.
2. **Аварийно-устойчивый журнал (write-ahead).** Append-only файл сессии,
   fsync при ключевых событиях: старт сессии (метаданные), завершение каждого
   сегмента (№, длительность, путь), пауза/возобновление, обрыв устройства,
   стоп. Цель — после внезапного завершения процесса однозначно восстановить
   состояние сессии и список готовых сегментов.
3. **Восстановление при старте.** При запуске обнаружить незавершённую сессию
   по журналу → `Recovering`: «починить» последний сегмент (усечённый WAV
   читаем как PCM — переписать заголовок по фактической длине данных), отметить
   сессию восстановленной, не потеряв более `flush_interval_ms` звука.
   Решение оператору: продолжить эту сессию или закрыть как восстановленную.
4. **Watchdog + авто-рестарт.** Контроль «живости» конвейера захвата
   (поступление семплов); при зависании дольше `watchdog_timeout_ms` —
   перезапуск потока захвата с продолжением записи в новый сегмент, событие в
   журнал и UI. Запись не должна молча «вставать».
5. **Отказ устройства.** Отключение/пропажа устройства не роняет сессию:
   переход в `Paused` (с пометкой `device_lost`), индикация; при возврате
   устройства и `device_reconnect.auto_resume=true` — авто-возобновление.
6. **Контроль ресурсов.** Мониторинг свободного места: при `disk_low_threshold_mb`
   — предупреждение в UI; при `disk_critical_mb` — защитные действия (стоп с
   гарантированным флашем + явное предупреждение, не потеря данных).
7. **Дублирующая дорожка (опц.).** При `mirror.enabled=true` — параллельная
   запись сегментов на `mirror.path` (второй носитель); сбой зеркала не влияет
   на основную запись (best-effort, логируется).
8. **Длина сессии.** При `max_session_hours` — предупреждение и/или авто-
   разбиение на под-сессии без разрыва звука.
9. **Тесты.** Kill-тест (убить процесс в ходе записи → рестарт → сессия
   восстановлена, потеря ≤ flush-интервала, сегменты валидны); симуляция
   обрыва устройства (пауза → возврат → авто-резюм); симуляция нехватки места
   (пороги срабатывают, данные целы); watchdog (искусственное зависание →
   рестарт). Юнит на переходы автомата и парсинг/реплей журнала.

## Шаги

1. `recorder/session.rs` — конечный автомат + события.
2. `recorder/journal.rs` — append-only журнал, fsync-политика, реплей при старте.
3. `recorder/recovery.rs` — обнаружение и починка незавершённой сессии,
   ремонт WAV-заголовка последнего сегмента.
4. `reliability/watchdog.rs` — контроль живости + авто-рестарт захвата.
5. `reliability/device_monitor.rs` — отслеживание появления/пропажи устройства,
   интеграция с автоматом (pause/auto-resume).
6. `reliability/disk_monitor.rs` — пороги места + защитные действия.
7. `reliability/mirror.rs` — опц. дублирующая запись сегментов.
8. Интеграция в `ipc/` (события состояния/предупреждений) и тесты.

## Критерии приёмки (Acceptance)

- **Kill-тест проходит:** принудительное завершение процесса в произвольный
  момент записи → после рестарта сессия определяется как незавершённая,
  чинится, потеряно не более `flush_interval_ms`; все сегменты валидны.
- Отключение микрофона в ходе записи не роняет приложение: сессия в `Paused`
  (`device_lost`), при возврате — авто-возобновление.
- Watchdog перезапускает зависший захват без потери сессии.
- Пороги свободного места срабатывают; при критическом — корректный стоп с
  флашем, без потери уже записанного.
- Все пороги/таймауты — из конфигурации; «магических чисел» в логике нет.
- `cargo clippy` чисто; тесты зелёные.

## Вне объёма этапа

- SHA-256/хеш-цепочка, шифрование at-rest, полный манифест, ретеншн — этап `03`.
- UI состояний/предупреждений сверх событий — этап `04` (здесь — события +
  ручная проверка).
- Выгрузка на сервер и оффлайн-очередь — этап `06`.

## Решено

- **Восстановление:** по умолчанию **дописывать ту же сессию** (заседание —
  одна логическая запись), помечая её «восстановленной» в журнале; оператору
  оставить выбор в UI (продолжить / закрыть как восстановленную).
- **Критическое заполнение диска:** **корректный стоп с гарантированным
  флашем** (без потери записанного). Аварийное переключение на `mirror.path`
  — только если зеркало включено (там уже есть копия); самопроизвольно на
  зеркало не переключаемся.
