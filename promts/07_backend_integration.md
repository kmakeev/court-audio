# Этап 07 — Приём записей в `ex_system` (Django)

> **Исполняется в репозитории `~/Progs/ex_system/`**, а не в `court-audio`.
> Цель — серверная сторона приёма аудиозаписей от модуля «Аудиопротокол»:
> регистрация сессии, возобновляемая чанковая загрузка, верификация
> целостности, привязка к делу и передача в обрабатывающий пайплайн W2.11.
> Этот этап **определяет контракт API**, на который опирается клиентский этап
> [`06_sync_agent.md`](06_sync_agent.md). Делать **до или синхронно** с 06.

## Результат этапа (Deliverables)

   - `source` — признак «capture-module» (vs ручная загрузка в W2.11),
   - `file` / `duration` / `sample_rate` / `channels` / `codec`,
   - `status` (зеркало `UploadResult.status`: PENDING/PROCESSING/SUCCESS/FAILED)
     и `extraction_status` (для последующего ASR/NER пайплайна W2.11),
   - `segments` (JSON: список сегментов с длительностью и SHA-256),
   - `integrity_hash` (хеш-цепочка по сегментам), `integrity_verified` (bool),
   - метаданные станции/оператора, `created_at` / `uploaded_at` / `finished_at`,
   - поля ретеншна (срок жизни локально/на сервере).
   Миграция; согласовать имена полей с тем, что ждёт W2.11.
2. **Эндпоинты приёма** (под существующим JWT, user-isolation как у
   `UploadResult`):
   - `POST /audio/sessions/` — регистрация сессии записи: дело (№/ФИО или
     `adjudication_id`), параметры формата/каналов, инфо станции →
     `{recording_id}` (+, при необходимости, токен/параметры загрузки).
   - **Чанковая возобновляемая загрузка** (init/part/complete):
     - `POST /audio/recordings/<id>/upload/init/` — заявить сегменты и их
       размеры/хеши,
     - `PUT  /audio/recordings/<id>/upload/part/<n>/` — приём части (идемпотентно
       по `n`, поддержка докачки),
     - `POST /audio/recordings/<id>/upload/complete/` — финализация: сборка,
       сверка состава сегментов.
   - `POST /audio/recordings/<id>/verify/` — сервер пересчитывает SHA-256 по
     сегментам и хеш-цепочку, сравнивает с заявленными → `integrity_verified`.

     метаданные.
   - `GET /audio/recordings/` / `GET /audio/recordings/<id>/` — список/деталь
     (для UI W2.11 и для статус-поллинга станции), статусы — как у `UploadResult`.
3. **Передача в пайплайн W2.11**: по завершении приёма + успешной верификации —

4. **Ретеншн**: `purge_expired_audio` по образцу `purge_expired_uploads`

5. **Аутентификация**: существующий JWT. Решено — **оператор обязан
   авторизоваться** (вход для старта; идущая запись переживает истечение

   `station_id` + **обязательный** `operator_id`. Зафиксировать в `docs/auth.md`.

Согласовать модель «зал/суд → дела»: как станция определяет свой зал (из

## Контракт API (черновик — синхронизировать с этапом 06)

- **Транспорт:** HTTPS, JWT в заголовке/cookie (как в `ex_system`).
- **Возобновляемость:** загрузка идёт сегментами; повтор `part/<n>` идемпотентен;
  `complete` проверяет, что все заявленные сегменты приняты и совпали по хешу.
  Это и есть устойчивость к обрывам сети («оффлайн-захват → догрузка»).
- **Целостность:** клиент шлёт SHA-256 каждого сегмента и хеш-цепочку; сервер
  пересчитывает в `verify`. Несовпадение → `integrity_verified=false` +
  читаемая ошибка (без падения приёма).
- **Идемпотентность сессии:** повторная регистрация той же сессии (по
  station-side UUID) не плодит дубли (как `get_or_create` у `UploadResult`).
- **Многоканал (этап 09):** часть адресуется по дорожке
  (`part/<track_id>/<part_index>`), `init` заявляет `tracks[]` с ролями, `verify`
  считает целостность per-track. Полностью — в разделе «Расширение под
  многоканал по ролям» ниже (одноканальный клиент = трек `single`).
- **Живая разметка (этап 10):** `init` дополнительно несёт `annotations`
  (закладки/интервалы ролей + хеш-цепочный лог); сервер сохраняет их, проверяет
  цепочку разметки и пробрасывает как подсказки в W2.11. Полностью — в разделе
  «Расширение под живую разметку» ниже (отсутствие `annotations` = запись без
  разметки).

## Расширение под многоканал по ролям (этап 09)

> Клиентская часть **уже реализована** в `court-audio` (этап 09): станция пишет
> N дорожек по ролям и выгружает их по расширенному контракту ниже. Этот раздел —
> **единый источник истины** для серверной реализации в `ex_system`; правится
> синхронно с [`06_sync_agent.md`](06_sync_agent.md) и
> [`../docs/configuration.md`](../docs/configuration.md) («Мультитрек-контракт»).
>
> Развилки согласованы с заказчиком: **одна `AudioRecording` на сессию + N
> треков внутри**; **справочник ролей — настраиваемый** (судебные дефолты).

### Принцип: сессия — единица, дорожка — часть

Одна `AudioRecording` (один `recording_id`, её потребляет W2.11) содержит **N
треков**. Одноканальная (v1) запись — частный случай: **один трек `track_id = 0`
роль `single`**. Сервер, не получивший `tracks`, обязан трактовать запись как
единственный трек `0` — так старый клиент работает без изменений.

### Модель: `AudioRecording` + `AudioTrack`

- **`AudioRecording`** (как в этапе 07) + поле `track_count` (int, дефолт 1).
  `integrity_verified` — агрегат (см. `verify`).
- **`AudioTrack`** (новая; миграция):
  - `FK AudioRecording`, `track_id` (int, 0-based), `role` (str), `label` (str),
  - `channel_index` (int), `source_device` (str, null),
  - `segments` (JSON: список `{index, sha256, chain_link, size_bytes, frames,
    started_at_unix_ms}`),
  - `integrity_hash` (финальное звено цепочки **этой дорожки**),
  - `integrity_verified` (bool),
  - `file`/`duration` — пофайловый результат сборки дорожки.
  - Уникальность `(recording, track_id)`.
- Роль — **свободная строка из справочника станции** (`audio.roles`), сервер её
  не валидирует по закрытому перечню (справочник настраиваемый), а **пробрасывает
  как приор говорящего** в диаризацию W2.11.

### Эндпоинты — трек-aware

- `POST /audio/sessions/` — `SessionMeta` дополнен **`track_count`** (роли — в
  `init`). Идемпотентность по `session_id` без изменений.
- `POST /audio/recordings/<id>/upload/init/` — тело **`{ "tracks": [TrackEntry…] }`**
  (было `{ "segments": […] }`). `TrackEntry = { track_id, role, label,
  final_chain_link, segments: [SegmentEntry…] }`; `SegmentEntry` несёт `track_id`.
  Сервер создаёт `AudioTrack` по каждому элементу и заявленный состав сегментов.
- `PUT /audio/recordings/<id>/upload/part/<track_id>/<part_index>/` — приём части
  **дорожки** (было `/part/<n>/`). Идемпотентно по `(track_id, part_index)`,
  докачка — по неотправленным парам.
- `POST /audio/recordings/<id>/upload/complete/` — сборка пофайловых дорожек,
  сверка состава по каждому треку.
- `POST /audio/recordings/<id>/verify/` — сервер пересчитывает SHA-256 и
  хеш-цепочку **по каждой дорожке** отдельно, пишет `AudioTrack.integrity_verified`
  и `integrity_hash`; ответ `{ "integrity_verified": bool }` — **агрегат** (=
  `all(track.integrity_verified)`), для совместимости с клиентским
  `VerifyOutcome`. Пер-трек детали — в `GET /audio/recordings/<id>/`.
- `GET /audio/recordings/<id>/` — в детали добавить `tracks[]` (роль + пер-трек
  статус целостности) для UI W2.11.

### Формат манифеста (для сверки в `verify`)

`manifest_version = 2` (было 1). Верхнеуровневый плоский `segments[]` **заменён**
на `tracks[]`; сегменты живут внутри трека. Источник структуры на станции —
`src-tauri/src/store/export.rs` (`RecordingManifest`/`TrackEntry`/`SegmentEntry`);
серверный разбор держать совместимым с ним.

### Передача в W2.11

По успешной верификации — как в п. 3 этапа 07, но в пайплайн уходят **пофайловые
дорожки + карта ролей** (вход диаризации: каждая дорожка — известный говорящий).
Мастер-даунмикс (опц. `audio.master_downmix`) — как единый обзорный файл, если
станция его прислала.

### Тесты (`criminal/tests/`) — дополнительно к этапу 07

- Мультитрек `init` → `part` ×N по разным трекам (вкл. повтор части) → `complete`
  → `verify(verified=true)`; роли доходят в пайплайн.
- **Изоляция целостности**: подмена сегмента одной дорожки → её
  `integrity_verified=false`, другие дорожки `true`; агрегат `false`.
- Обратная совместимость: одноканальный клиент (без `tracks`) принимается как
  трек `0` роль `single`.
- Идемпотентность `part/<track_id>/<part_index>`.

### Обратная совместимость и миграция

- Существующие записи → один `AudioTrack` `track_id=0` роль `single` (backfill в
  миграции), `track_count=1`.
- Эндпоинты принимают и старую (`segments`), и новую (`tracks`) форму `init` —
  первую как один трек `0`.

## Расширение под живую разметку (этап 10)

> Клиентская часть **уже реализована** в `court-audio` (этап 10): станция ведёт
> ручную живую разметку заседания — закладки с таймкодом (категория + опц.
> комментарий) и интервалы «сейчас говорит <роль>» — и выгружает её вместе с
> записью. Этот раздел — **единый источник истины** для серверной реализации в
> `ex_system`; правится синхронно с [`06_sync_agent.md`](06_sync_agent.md),
> [`10_markers_realtime.md`](10_markers_realtime.md) и
> [`../docs/configuration.md`](../docs/configuration.md) («Контракт разметки»).
>
> Развилки согласованы с заказчиком: **разметка ручная** (ASR/диаризацию не
> дублируем — это W2.11); **категории закладок — настраиваемый справочник
> станции** (`markers.categories`); **роли интервалов — из того же справочника
> `audio.roles`**, что и дорожки (этап 09).

### Принцип: разметка — подсказки W2.11 под хеш-цепочкой

Разметка на станции — **append-only лог действий** (постановка/правка/удаление
метки, начало/конец интервала роли), под собственной хеш-цепочкой
(tamper-evident, с авторством). На сервер приходит и **свёрнутое текущее
состояние** (метки/интервалы — прямой вход для протокола/диаризации W2.11), и
**полный лог** (для серверной проверки целостности разметки). Сервер разметку
**не генерирует и не редактирует** — только принимает, верифицирует и
пробрасывает в пайплайн.

### Модель: разметка при `AudioRecording`

- Хранить при записи: **свёрнутые** `markers[]` и `role_spans[]` (вход W2.11) +
  **лог** действий и `annotations_chain_final` (для верификации). Форма —
  согласовать с W2.11 (нормализованные `AudioMarker`/`AudioRoleSpan` + JSON-лог,
  либо единый JSON-блок `annotations`); имена полей свести с тем, что заложит
  реализация W2.11 (как и для `AudioTrack`).
- `annotations_verified` (bool) — **отдельный** флаг целостности разметки (не
  сворачивается в пер-трековый `integrity_verified`, чтобы не менять семантику
  `VerifyOutcome`); детали — в `GET /audio/recordings/<id>/`.
- Смещения — семпл-точные по общей оси сессии (`offset_samples` + `offset_ms`);
  сервер использует их как таймкоды подсказок.

### Эндпоинты — annotation-aware

- `POST /audio/recordings/<id>/upload/init/` — тело расширено до
  **`{ "tracks": [TrackEntry…], "annotations": AnnotationsExport }`**, где
  `AnnotationsExport = { chain_final_link, markers[], role_spans[], log[] }`:
  - `markers[] = { id, category, comment?, offset_samples, offset_ms,
    operator_id, at_unix_ms }`,
  - `role_spans[] = { id, role, start_offset_samples, start_offset_ms,
    end_offset_samples?, end_offset_ms?, operator_id, at_unix_ms }`,
  - `log[] = { seq, action, target_id, category?, role?, comment?,
    offset_samples, offset_ms, operator_id, at_unix_ms, chain_link }`,
    `action ∈ { marker_added, marker_edited, marker_removed, role_started,
    role_ended }`.
  Сервер сохраняет разметку записи. Приём **идемпотентен** (повторный `init`
  той же записи не плодит дублей — как и для дорожек).
- `POST /audio/recordings/<id>/verify/` — помимо per-track целостности,
  пересчитать **хеш-цепочку разметки** по `log[]` и сравнить с `chain_link`
  каждой записи и `chain_final_link`; выставить `annotations_verified`. Пустая
  разметка → `annotations_verified = true`.
- `GET /audio/recordings/<id>/` — в детали добавить свёрнутую разметку (метки/
  интервалы) и `annotations_verified` для UI W2.11.

### Формат манифеста (для сверки в `verify`)

`manifest_version = 3` (было 2): к `tracks[]` добавлен блок `annotations`.
Источник структуры на станции —
`src-tauri/src/store/export.rs` (`RecordingManifest`/`AnnotationsExport`);
модель/цепочка/канонический дайджест — `src-tauri/src/integrity/annotations.rs`,
серверный пересчёт держать байт-совместимым с ним:

- канонический дайджест записи = `SHA-256` от полей `action`, `target_id`,
  `category`, `role`, `comment`, `offset_samples`, `offset_ms`, `operator_id`,
  `at_unix_ms`, соединённых разделителем `U+001F` (unit separator; пустое
  поле — пустая строка);
- звено `chain_link[i] = SHA-256(hex(chain_link[i-1]) ‖ hex(digest_i))`,
  genesis (`i = 0`) = `SHA-256(hex(digest_0))` — та же схема, что у сегментной
  цепочки (hex-строки), поэтому переиспользуется серверный код проверки цепочки.

### Передача в W2.11

По успешной верификации в пайплайн уходят **свёрнутые метки и интервалы ролей**
как подсказки: `role_spans[]` — сильный приор диаризации (известный говорящий на
интервале, дополняет пер-трековые роли этапа 09), `markers[]` — таймкодовые
закладки для чернового протокола (границы допроса, инциденты и т.п.). Категории
и комментарии — свободные строки (справочник станции), сервер их не
валидирует по закрытому перечню, а пробрасывает как есть.

### Тесты (`criminal/tests/`) — дополнительно к этапу 07

- `init` с `annotations` → `verify` → `annotations_verified=true`; метки и
  интервалы ролей доходят в пайплайн W2.11.
- **Tamper разметки:** подмена поля записи лога (категория/смещение/автор) →
  `annotations_verified=false`, при этом per-track `integrity_verified` не
  затронут (флаги независимы).
- Обратная совместимость: `init` без `annotations` (одноканальный/легаси-клиент,
  `manifest_version ≤ 2`) принимается; `annotations_verified=true` (пусто).
- Идемпотентность повторного `init` с той же разметкой.

### Обратная совместимость и миграция

- Существующие записи → пустая разметка (`markers=[]`, `role_spans=[]`,
  `annotations_verified=true`); поля/таблицы разметки — с дефолтами (миграция
  аддитивна).
- Эндпоинт `init` принимает тело и без `annotations` (как `manifest_version = 2`).

## Критерии приёмки (Acceptance)

- Полный цикл проходит в тестах: `sessions/` → `upload/init` → `part` ×N (вкл.
  повтор части) → `complete` → `verify` (verified=true) → привязка к делу →
  запись видна в списке со статусом, готовым к обработке W2.11.
- Tamper-кейс (изменённый сегмент) даёт `integrity_verified=false`.
- **Многоканал:** мультитрек-цикл проходит; подмена сегмента одной дорожки не
  ломает целостность других; роли доходят до W2.11; одноканальный клиент
  принимается как трек `single`.
- **Разметка (этап 10):** `init` с `annotations` принимается, цепочка разметки
  верифицируется (`annotations_verified`), метки/интервалы ролей доходят до
  W2.11; подмена записи лога даёт `annotations_verified=false` без влияния на
  per-track целостность; клиент без `annotations` принимается (пустая разметка).
- `pytest` зелёный, без регрессий golden cases (ядро расчёта не трогаем).
- Эндпоинты задокументированы в `docs/rest_api.md`; миграция применяется.

## Вне объёма этапа

- Сама ASR/диаризация и NER-апплай — это W2.11 (`ex_system`), здесь только
  запуск пайплайна и статусная модель.
- Клиентская выгрузка со станции — этап [`06_sync_agent.md`](06_sync_agent.md).
- ГОСТ-подпись/КриптоПро на сервере — фаза 2 (`11_gost_signing.md`).

## Решено

- Авторизация: оператор обязателен + идентичность станции (см. п. 5).
- Серверный ретеншн — `AUDIO_RETENTION_HOURS` (дефолт 30 сут), зеркало
  `purge_expired_uploads`; параметры — [`../docs/configuration.md`](../docs/configuration.md).
- **Мультитрек-контракт (этап 09):** одна `AudioRecording` на сессию + N треков
  (`AudioTrack`), а не отдельная запись на дорожку. `manifest_version = 2`,
  per-track целостность, `part/<track_id>/<part_index>`. См. раздел «Расширение
  под многоканал по ролям».
- **Справочник ролей:** настраиваемый на станции (`audio.roles`, судебные
  дефолты); сервер роли не валидирует по закрытому перечню, а пробрасывает в
  диаризацию W2.11.
- **Разметка-контракт (этап 10):** `manifest_version = 3`; `init` несёт блок
  `annotations` (свёрнутые метки/интервалы ролей + хеш-цепочный лог); `verify`
  выставляет отдельный `annotations_verified`; разметка — вход подсказок W2.11
  (диаризация/протокол). См. раздел «Расширение под живую разметку». Категории
  закладок — настраиваемый справочник станции (`markers.categories`), сервер их
  не валидирует, а пробрасывает как есть.

## Открытые вопросы к заказчику

- Имена/состав полей `AudioRecording`/`AudioTrack` — сверить с тем, что заложит
  реализация W2.11, чтобы модель была единой (особенно `role`/`label` как вход
  диаризации).
- Форма хранения разметки на сервере (нормализованные `AudioMarker`/
  `AudioRoleSpan` + JSON-лог vs единый JSON-блок `annotations`) и как W2.11
  потребляет метки/интервалы ролей в черновике протокола — сверить с реализацией
  W2.11.
- Связь серверного ретеншна с фактом привязки записи к делу (удалять ли
  непривязанные раньше).
