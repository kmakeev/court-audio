# Этап 07 — Приём записей в `ex_system` (Django)

> **Исполняется в репозитории `~/Progs/ex_system/`**, а не в `court-audio`.
> Цель — серверная сторона приёма аудиозаписей от модуля «Аудиопротокол»:
> регистрация сессии, возобновляемая чанковая загрузка, верификация
> целостности, привязка к делу и передача в обрабатывающий пайплайн W2.11.
> Этот этап **определяет контракт API**, на который опирается клиентский этап
> [`06_sync_agent.md`](06_sync_agent.md). Делать **до или синхронно** с 06.

## Результат этапа (Deliverables)

1. **Модель `AudioRecording`** (новая либо согласованная с заложенной в W2.11):
   - `FK Adjudication (null=True)` — привязка к делу (может прийти позже),
   - `source` — признак «capture-module» (vs ручная загрузка в W2.11),
   - `file` / `duration` / `sample_rate` / `channels` / `codec`,
   - `status` (зеркало `UploadResult.status`: PENDING/PROCESSING/SUCCESS/FAILED)
     и `extraction_status` (для последующего ASR/NER пайплайна W2.11),
   - `segments` (JSON: список сегментов с длительностью и SHA-256),
   - `integrity_hash` (хеш-цепочка по сегментам), `integrity_verified` (bool),
   - метаданные станции/оператора, `created_at` / `uploaded_at` / `finished_at`,
   - поля ретеншна (срок жизни локально/на сервере).
   Миграция; согласовать имена полей с тем, что ждёт W2.11.
2. **Эндпоинты приёма** (под существующим JWT, user-isolation как у
   `UploadResult`):
   - `POST /audio/sessions/` — регистрация сессии записи: дело (№/ФИО или
     `adjudication_id`), параметры формата/каналов, инфо станции →
     `{recording_id}` (+, при необходимости, токен/параметры загрузки).
   - **Чанковая возобновляемая загрузка** (init/part/complete):
     - `POST /audio/recordings/<id>/upload/init/` — заявить сегменты и их
       размеры/хеши,
     - `PUT  /audio/recordings/<id>/upload/part/<n>/` — приём части (идемпотентно
       по `n`, поддержка докачки),
     - `POST /audio/recordings/<id>/upload/complete/` — финализация: сборка,
       сверка состава сегментов.
   - `POST /audio/recordings/<id>/verify/` — сервер пересчитывает SHA-256 по
     сегментам и хеш-цепочку, сравнивает с заявленными → `integrity_verified`.
   - `PATCH /audio/recordings/<id>/` — привязка/смена `adjudication_id`,
     метаданные.
   - `GET /audio/recordings/` / `GET /audio/recordings/<id>/` — список/деталь
     (для UI W2.11 и для статус-поллинга станции), статусы — как у `UploadResult`.
3. **Передача в пайплайн W2.11**: по завершении приёма + успешной верификации —
   перевод `extraction_status` в очередь и запуск Celery-таска ASR/NER (тот же
   паттерн, что `process_pdf_task` → `extract_entities_task`). Если ASR-сервис
   ещё не подключён — оставить статус «ожидает обработки» + точку расширения.
4. **Ретеншн**: `purge_expired_audio` по образцу `purge_expired_uploads`
   (Celery beat, удаление файла + каскад метаданных по окну хранения; решение
   о связи с фактом привязки к делу — согласовать).
5. **Аутентификация**: существующий JWT. Решено — **оператор обязан
   авторизоваться** (вход для старта; идущая запись переживает истечение
   токена; оффлайн-старт по кэшированной сессии — клиентская часть в `06`).
   У станции — своя учётка на зал (транспорт выгрузки). В `AudioRecording` —
   `station_id` + **обязательный** `operator_id`. Зафиксировать в `docs/auth.md`.
6. **Тесты** (`criminal/tests/`): регистрация сессии, чанковая загрузка с
   докачкой, верификация целостности (happy + tamper), привязка к делу,
   user-isolation, переход статусов, ретеншн-purge.
7. **Документация**: `docs/rest_api.md` (новые эндпоинты + формат),
   `docs/data_model.md` (модель), при необходимости `docs/celery_tasks.md`.

## Список дел для станции (для `05`)

Станция привязывает запись к делу из **докета суда/зала** (решение заказчика),
а не из личных дел оператора. Нужен **slim-эндпоинт** списка дел:

- скоуп — докет суда/зала станции (фильтр по суду/залу, не по владельцу/
  assignee), минимальные поля (`id`, № дела, ФИО подсудимых, дата);
- permission — «станция/оператор читает докет своего зала» (новая ветка;
  существующий user-scoped `/adjudications/` недостаточен);
- ПДн-минимизация: только поля для привязки, лимит/пагинация.

Согласовать модель «зал/суд → дела»: как станция определяет свой зал (из
идентичности станции).

## Контракт API (черновик — синхронизировать с этапом 06)

- **Транспорт:** HTTPS, JWT в заголовке/cookie (как в `ex_system`).
- **Возобновляемость:** загрузка идёт сегментами; повтор `part/<n>` идемпотентен;
  `complete` проверяет, что все заявленные сегменты приняты и совпали по хешу.
  Это и есть устойчивость к обрывам сети («оффлайн-захват → догрузка»).
- **Целостность:** клиент шлёт SHA-256 каждого сегмента и хеш-цепочку; сервер
  пересчитывает в `verify`. Несовпадение → `integrity_verified=false` +
  читаемая ошибка (без падения приёма).
- **Идемпотентность сессии:** повторная регистрация той же сессии (по
  station-side UUID) не плодит дубли (как `get_or_create` у `UploadResult`).

## Критерии приёмки (Acceptance)

- Полный цикл проходит в тестах: `sessions/` → `upload/init` → `part` ×N (вкл.
  повтор части) → `complete` → `verify` (verified=true) → привязка к делу →
  запись видна в списке со статусом, готовым к обработке W2.11.
- Tamper-кейс (изменённый сегмент) даёт `integrity_verified=false`.
- `pytest` зелёный, без регрессий golden cases (ядро расчёта не трогаем).
- Эндпоинты задокументированы в `docs/rest_api.md`; миграция применяется.
