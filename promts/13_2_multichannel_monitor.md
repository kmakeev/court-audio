# Этап 13.2 — Исправление R-008: многоканальный мониторинг уровня до записи

> Проект **«Аудиопротокол»**, исправление по бэклогу
> ([`../docs/backlog.md`](../docs/backlog.md), R-008). Живой мониторинг уровня
> **до старта** должен покрывать все дорожки многоканальной карты — как это уже
> делает сама запись, — чтобы оператор мог проверить каждый микрофон (в т.ч.
> «дубли» одного источника на разных дорожках) до начала заседания.

## Контекст

Сейчас превью-мониторинг — **одиночный** поток `MonitorSession` по единственному
устройству `settings.audio.device` с жёстким `track_id: 0`
([`src-tauri/src/ipc/audio_cmds.rs:664-678`](../src-tauri/src/ipc/audio_cmds.rs#L664)
`build_params`; `start_monitor` → `MonitorSession::start` в
[`src-tauri/src/audio/capture.rs:437`](../src-tauri/src/audio/capture.rs#L437)).
Он эмитит уровни только под `track_id 0`, а UI `TrackMeters`
(`src/screens/Record.tsx:977`) рисует метр на **каждую** сконфигурированную дорожку —
поэтому все дорожки, кроме 0-й (включая «дубли» микрофона), до записи молчат. При
записи же поднимается многодорожечная ветка (`MultiTrackSession`), где каждая
дорожка открывает своё устройство и эмитит свой `track_id`
([`ipc/audio_cmds.rs:315-322`](../src-tauri/src/ipc/audio_cmds.rs#L315)) — отсюда
расхождение «до» vs «во время».

Параметры устройств/дорожек — карта дорожек (`recorder::multitrack`), настройки
`audio.tracks`/`audio.multichannel`. Пороги/частоты — реестр
[`../docs/configuration.md`](../docs/configuration.md).

## Результат этапа (Deliverables)

1. **Многодорожечный монитор.** В многоканальном режиме `start_monitor` поднимает
   превью-поток **на каждую дорожку/устройство** карты дорожек и эмитит уровни под
   их `track_id` (как ветка записи, но без сегментного райтера/журнала/зеркала).
   Одноканальный режим — как сейчас (один поток, `track_id 0`).
2. **Переиспользование карты дорожек.** Источник дорожек монитора — тот же, что у
   записи (та же резолвинг-логика устройств/каналов), чтобы картинка «до» точно
   совпадала с «во время», включая дубли одного источника на разных дорожках.
3. **Корректная остановка/перезапуск.** Все превью-потоки гасятся и освобождают
   устройства при старте записи, смене настроек и уходе с экрана (нет утечек
   `cpal`-потоков; идемпотентный `stop_monitor`).
4. **Тесты.** Многоканальная карта (в т.ч. с дублирующимся устройством на 2+
   дорожках) → монитор эмитит события под всеми `track_id`; одноканал — один
   `track_id 0`; старт записи корректно снимает монитор; повторный `start_monitor`
   не плодит потоки.

## Шаги

1. Вынести резолвинг дорожек/устройств монитора в общий с записью путь.
2. `MonitorSession` → много-дорожечный вариант (по потоку на дорожку) либо
   `MultiMonitorSession`, эмиссия под `track_id` дорожки.
3. `start_monitor`/`stop_monitor`: поднять/погасить набор потоков; согласовать с
   переходом в запись (монитор не конфликтует с `MultiTrackSession`).
4. Тесты (юнит на эмиссию `track_id`, интеграционный на старт/стоп).

## Критерии приёмки (Acceptance)

- До записи в многоканале индикаторы **всех** дорожек живые, включая «дубли» одного
  микрофона на разных дорожках; поведение «до» совпадает с «во время».
- Старт записи снимает превью-потоки без конфликта за устройство; нет утечек
  потоков при повторных заходах на экран.
- Одноканальный путь не изменился; зелёный CI.
