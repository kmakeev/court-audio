# Этап 11 — ГОСТ ЭЦП записей (КриптоПро) (фаза 2)

> Проект **«Аудиопротокол»**, фаза 2. Цель — придать записям **юридически
> значимую целостность**: подпись сегментов/манифеста по ГОСТ средствами
> **КриптоПро** поверх уже имеющихся SHA-256 + хеш-цепочки (`03`). Закрывает
> строку таблицы «Целостность: SHA-256 + метки + журнал (v1) → ГОСТ ЭЦП
> (КриптоПро) (фаза 2)».

## Контекст

Архитектура и конфигурация — [`../CLAUDE.md`](../CLAUDE.md) (раздел «ПДн и
безопасность», таблица целостности). Параметры — реестр
[`../docs/configuration.md`](../docs/configuration.md) (раздел «Целостность»:
`integrity.gost_sign` = `false` → включается здесь; новые параметры — в реестр).
Код — `src-tauri/integrity/` (+ `store/`, `sync/`). Опирается на хеш-цепочку и
журнал (`03`) и серверную верификацию (`06`/`07`).

> Требует лицензий/сертификатов КриптоПро и ключевых носителей — организационная
> предпосылка. Импортозамещение: сертифицированная российская криптография в
> тракте подписи, **без зарубежных крипто-компонентов**.

## Результат этапа (Deliverables)

1. **Подпись по ГОСТ.** Подпись корня хеш-цепочки / манифеста сессии (и/или
   сегментов) по **ГОСТ Р 34.10-2012** (подпись) на базе **ГОСТ Р 34.11-2012**
   («Стрибог»), средствами **КриптоПро CSP**. Формат подписи — согласовать
   (CAdES/откреплённая PKCS#7) с приёмной стороной.
2. **Интеграция КриптоПро CSP** на целевых ОС (Astra SE / РЕД ОС / Windows),
   сертифицированные сборки; ключ — с носителя (Рутокен/JaCarta/…)/HSM.
   Абстракция провайдера подписи (seam), чтобы не зашивать вендор в логику.
3. **Метка времени (опц.).** Доверенная метка через TSP по ГОСТ, если сервис
   доступен в контуре суда (адрес — настройка).
4. **Подписанный журнал/цепочка custody.** Журнал событий (`03`) и цепочка
   владения подписываются; фиксация «кто/каким сертификатом/когда».
5. **Серверная верификация.** `ex_system` проверяет ГОСТ-подпись при приёме
   (`integrity_verified` учитывает подпись); расширение контракта `07`;
   `integrity.gost_sign=true` включает тракт.
6. **Ретеншн с учётом подписи.** Удаление локальной копии — только после
   подтверждённой **подписанной** целостности на сервере (согласовать с `03`).
7. **Тесты.** Подпись/проверка на тестовых сертификатах; корректная привязка
   подписи к корню хеш-цепочки; отрицательные тесты (подмена сегмента →
   подпись/цепочка не сходится); поведение без носителя/сертификата (не терять
   данные, статус ошибки); серверная верификация подписи.

## Шаги

1. `integrity/gost.rs` — seam провайдера подписи (КриптоПро CSP), выбор
   сертификата/носителя из конфигурации.
2. Подпись корня цепочки/манифеста; формат (CAdES/PKCS#7); опц. TSP-метка.
3. Расширение манифеста (`03`): поля подписи (сертификат, время, значение).
4. `sync/` — передача подписи серверу; расширение контракта `07`; учёт в
   `verify` (`06`).
5. Ретеншн (`03`): условие удаления — подтверждённая подписанная целостность.
6. Тесты (позитив/негатив, тестовые сертификаты).

## Критерии приёмки (Acceptance)

- Запись подписывается по ГОСТ (КриптоПро); подпись привязана к корню хеш-
  цепочки/манифесту; журнал/цепочка custody подписаны.
- Подмена любого сегмента ломает проверку (цепочка + подпись); статус ошибки.
- `ex_system` проверяет ГОСТ-подпись при приёме; `integrity.gost_sign=true`
  включает тракт; ретеншн учитывает подписанную целостность.
- Отсутствие носителя/сертификата не приводит к потере данных (накопление +
  статус).
- Параметры/сертификаты — из конфигурации/носителя, не хардкод; `cargo clippy`
  чисто; тесты зелёные (на тестовых ключах).

## Вне объёма этапа

- Шифрование at-rest и SHA-256/хеш-цепочка — уже в `03` (здесь — подпись поверх).
- ASR/NER/протокол — в `ex_system` (W2.11).
- Многоканал (`09`), метки (`10`), видео (`12`) — подпись применяется и к их
  артефактам, но сами функции — в своих этапах.

## Решено

- v1 — только SHA-256 + хеш-цепочка + метки времени + журнал (без ЭЦП). ГОСТ ЭЦП
  — фаза 2 (эта задача), требует КриптоПро и ключевых носителей.

## Открытые вопросы

- **Продукт/версия КриптоПро** (CSP, редакция), сертификация под целевые ОС —
  от заказчика.
- **Ключевые носители** (Рутокен/JaCarta/HSM) и модель хранения ключа станции/
  оператора.
- **Что именно подписываем** (только корень манифеста vs каждый сегмент) и
  **формат подписи** (CAdES-BES/-T, откреплённый PKCS#7) — согласовать с приёмной
  стороной; синхронно править `07`.
- **Доступность TSP** по ГОСТ в контуре суда (адрес/сервис).
