## Часть A — B-001: автономный офлайн-старт станции по PIN

### Суть

Сейчас офлайн-старт возможен **только после** хотя бы одного онлайн-входа в окне
`auth.operator.cached_session_hours` (вход кэширует сессию + хеш PIN, дальше
`auth_unlock_offline` по PIN). В **изолированном зале**, где `ex_system` вообще
недоступен, станцию **нельзя запустить впервые** — нет ни кэша сессии, ни
возможности войти онлайн. Нужен режим первого запуска **без обязательного
онлайн-входа**: разблокировка по PIN, заданному при провижининге; идентичность
оператора/станции — из провижининг-профиля, а не из онлайн-сессии. Выгрузка (когда
связь появится) — через уже существующую отложенную очередь.

### Ключевая идея реализации

B-001 = **распространить паттерн админ-PIN на операторскую идентичность**.
Отличие от админ-PIN: оператору нужна ещё **привязка к `operator_id`/`station_id`**
(админ-PIN идентичность не несёт). То есть провижининг-профиль оператора зала +
PIN (env/файл) → зашифрованный блоб ключом станции → офлайн-разблокировка,
идентичная по контуру `auth_unlock_offline`, но источник идентичности —
провижининг, а не `CachedSession` из онлайн-входа.

### Открытые вопросы — СНАЧАЛА согласовать с заказчиком (не зашивать)

Эти развилки меняют модель доступа; реализацию начинать **после** ответов
(`AskUserQuestion`/явное согласование), фиксируя решения в разделе «Решено» этого
файла и в [`../docs/auth.md`](../docs/auth.md):

1. **Источник идентичности офлайн.** Как получить `operator_id`/`station_id` без
   сервера: провижининг-файл на станции? предзаданный маппинг «PIN → оператор»?
   единый «сервисный оператор зала» (одна учётка на зал)?
2. **Согласование с решением «оператор обязан авторизоваться».** Автономный режим
   ослабляет требование онлайн-входа — нужно явное «да» заказчика и запись в
   `CLAUDE.md` (таблица «Зафиксированная конфигурация»), иначе это отступление от
   зафиксированного.
3. **Приём на сервере.** Как `ex_system` валидирует записи с «офлайн-провиженной»
   идентичностью при догрузке (`operator_id`/`station_id`, контракт `07`). Правка
   контракта — **синхронно** в `ex_system` и `promts/06_sync_agent.md`/
   `07_backend_integration.md` (правило `CLAUDE.md`).
4. **Докет дел офлайн.** Если онлайн-синхронизации не было ни разу — предзагружаемый/
   импортируемый `case_cache` (иначе привязка к делу недоступна).
5. **Область действия флага.** Автономный режим — только для явно
   провижиненных изолированных залов (новый ключ реестра, дефолт «выкл.»), чтобы
   обычные станции не ослабляли вход. Имя ключа — в `configuration.md`.

### Инварианты (независимо от ответов)

- Секрет PIN хранится **только хешем** (Argon2id), как в кэш-сессии/админ-PIN; в
  `settings.json` секретов нет (текущее правило) — переиспользовать `hash_pin`/
  `verify_pin` из `sync::auth`.
- Зашифрованный блоб — ключом станции (`store::crypto`), с той же fail-secure
  реакцией на отсутствие ключа, что введена этапом 13.5 (не молчать).
- Идущую запись изменения идентичности не прерывают (гейт — только на старте).

### Затрагивает

`sync/auth` (провижининг vs cached-session), `store::admin_pin` (образец),
`ipc/auth_cmds.rs` (новый путь разблокировки), `SessionMeta`/контракт `07`,
`configuration.md` (ключи провижининга + флаг автономного режима),
[`../docs/auth.md`](../docs/auth.md), [`../docs/packaging.md`](../docs/packaging.md)
(провижининг профиля оператора рядом с `COURT_AUDIO_ADMIN_PIN`), приём в
`ex_system`.

## Часть B — R-006: Windows compact overlay (белая рамка + подвисание)

### Дефект

Под Windows при активации «окно поверх» (`open_compact_overlay`) появляется
**белая рамка**, и приложение **перестаёт реагировать на старт/стоп** записи.
На macOS не воспроизводится (WebView2-специфика).

### Где смотреть / гипотезы

`open_compact_overlay` строит `WebviewWindowBuilder` (label `overlay`, URL
`index.html?window=overlay`, `always_on_top(true)`, `decorations(false)`,
`skip_taskbar(true)`). Проверить на Windows/WebView2:

- **Белая рамка (а):** WebView2 показывает белый фон до первого рендера; либо
  маршрут `?window=overlay` не отрисовывается отдельным окном на Windows; либо
  нужен явный фон окна/`transparent`. Проверить: фоновый цвет окна, порядок
  показа (создавать скрытым → показать после первого рендера), корректность
  роутинга overlay-окна во фронте (`main.tsx`/`CompactOverlay`).
- **Подвисание старт/стоп (б):** overlay как отдельное окно перехватывает
  фокус/ввод (`always_on_top`); либо команды записи завязаны на состояние/окно,
  недоступное из overlay; либо создание окна блокирует UI-поток. Проверить: что
  `CaptureState` — app-managed и **общий** для обоих окон (события `capture_state`
  доходят до обоих через `app.emit`); что кнопки старт/стоп в основном окне не
  теряют обработчики после открытия overlay; что билд окна не идёт синхронно на
  горячем пути.

### Ожидаемо

Overlay корректно рендерится на Windows (без белой рамки), управление записью
остаётся отзывчивым из основного окна и/или из overlay. Регресс на macOS/Linux —
поведение не хуже текущего.

### Затрагивает

`ipc/ui_cmds.rs` (`open_compact_overlay`/`close_compact_overlay`), фронт-маршрут
`?window=overlay` (`CompactOverlay`), WebView2 (Windows), `ui.compact_overlay`/
`ui.hall_mode` (этап 10.5/10.6), экран «Запись».

## Результат этапа (Deliverables)

1. **B-001 (после согласования):** автономный офлайн-старт по провижиненному
   PIN в изолированном зале — идентичность из провижининг-профиля, разблокировка
   офлайн, отложенная выгрузка; флаг автономного режима (дефолт «выкл.») в
   реестре; fail-secure к отсутствию ключа станции (наследует 13.5).
2. **R-006:** overlay на Windows без белой рамки; старт/стоп записи отзывчивы
   при открытом overlay.
3. **Доки.** `auth.md` (модель автономной идентичности + принятые ответы на
   открытые вопросы), `configuration.md` (новые ключи), `packaging.md`
   (провижининг профиля оператора). Контракт `07` — синхронно в `ex_system`, если
   B-001 его затрагивает.
4. **Тесты.** B-001: провижининг → офлайн-разблокировка по PIN без сети
   (чистая логика + store-слой, как у `admin_pin`); неверный PIN/нет ключа →
   fail-secure. R-006: где применимо — юнит на чистую логику окна/роутинга; сам
   рендер/фокус на Windows — по ручному чек-листу (нативное окно в CI без дисплея
   не исполняется, как трей).

## Критерии приёмки (Acceptance)

- **B-001:** в изолированном зале (нет сети/`ex_system`) станция стартует
  **впервые** по провижиненному PIN, оператор/станция идентифицированы, запись
  привязывается к делу (офлайн-докет) и уходит в отложенную очередь; секрет PIN —
  только хеш; открытые вопросы согласованы и записаны. Обычные станции без флага
  автономного режима — поведение не изменилось.
- **R-006:** на Windows overlay открывается без белой рамки, старт/стоп записи
  отзывчивы; на macOS/Linux регресса нет.
- Доки обновлены; зелёный CI (`cargo clippy`/`test`, `tsc`/`vitest`).
