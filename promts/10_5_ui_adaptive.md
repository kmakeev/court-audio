# Этап 10.5 — Дизайн и адаптивность интерфейса

> Проект **«Аудиопротокол»**, улучшения после v1
> ([`../docs/recommendations.md`](../docs/recommendations.md), п. 5). Цель —
> интерфейс, который аккуратно работает на любом мониторе зала (узкие окна,
> масштабирование ОС, тач-моноблоки) и в котором **состояние записи всегда на
> виду** — включая «режим зала» с крупным статусом.

## Контекст

Текущий макет фиксированный: боковая панель 232 px (`shell/AppShell.tsx`),
контент `maxWidth` 760–880, стили инлайновые, брейкпоинтов нет. Дизайн-система
— PravoUI через `.design-sync` (только токены, вручную CSS-переменные не
задавать — см. `.design-sync/conventions.md`). Параметры поведения — реестр
[`../docs/configuration.md`](../docs/configuration.md) (кандидаты:
`ui.hall_mode.*`, `ui.compact_overlay.enabled`).

## Результат этапа (Deliverables)

1. **Адаптивная сетка.** Брейкпоинты: на узких окнах боковая навигация
   сворачивается в иконки/бургер; формы и карточки перестраиваются; корректная
   работа при масштабировании ОС 100–150 % и от ~1280×720. Проверка тач-ввода
   (размеры целей нажатия).
2. **Глобальный индикатор записи.** Состояние записи (идёт/пауза/сбой) + хронометр
   видны из **любого** экрана: индикатор в шапке (`AppShell`) + иконка в
   системном трее с состоянием.
3. **Режим зала («большой статус»).** Разворачиваемая панель: крупный
   хронометр, состояние, уровень сигнала — читается с нескольких метров.
   Опционально — компактное окно «поверх всех окон» (Tauri multi-window),
   чтобы оператор работал в других программах, не теряя контроль записи.
4. **Гигиена стилей.** Повторяющиеся инлайн-стили (подписи полей, нейтральные
   кнопки, сетки карточек) — в разделяемые классы/компоненты поверх токенов
   PravoUI; никаких самодельных CSS-переменных. Задел под тёмную тему: только
   токены, при появлении темы в PravoUI — включение без переделки экранов.
5. **Доступность.** Контраст, видимые фокус-обводки, полная работа с
   клавиатуры (включая диалоги и мастера), aria-атрибуты (дополнить
   существующие), `prefers-reduced-motion` для анимаций метров.
6. **Тесты/проверка.** Смоук-тесты рендера на узких вьюпортах; ручной чек-лист
   по матрице (узкое окно / 150 % / тач) в `docs/`; трей и режим зала — смоук
   через фикстуры состояния.

## Шаги

1. Инвентаризация повторяющихся инлайн-стилей → разделяемые классы/компоненты.
2. Брейкпоинты сетки `AppShell` + перестройка экранов (Запись, Сессии,
   Настройки, Диагностика).
3. Глобальный индикатор записи в шапке + трей-иконка (Tauri tray API).
4. Режим зала: панель большого статуса; опц. компакт-окно поверх всех окон.
5. Проход по доступности.
6. Смоук-тесты + чек-лист проверки в `docs/`.

## Критерии приёмки (Acceptance)

- На окне ~1280×720 и при масштабировании ОС 150 % все экраны работоспособны
  без горизонтального скролла и обрезаний.
- Состояние записи видно с любого экрана приложения и из трея; режим зала
  читается с нескольких метров.
- Инлайн-стили ключевых паттернов вынесены; используются только токены PravoUI.
- Клавиатурная навигация полна; фокус видим; контраст в норме.
- Параметры режима зала/оверлея — из реестра; тесты и линт зелёные.

## Вне объёма этапа

- Собственная тёмная тема в обход PravoUI — не делаем (ждём токены DS).
- Новые функциональные экраны (плеер `10.1`, мастера `10.6`) — свои этапы;
  они наследуют адаптивную сетку этого этапа.
- Локализация (продукт русскоязычный).

## Решено

- Стилизация — только токенами/компонентами PravoUI (`.design-sync`), без
  ручных CSS-переменных: соглашение проекта.

## Открытые вопросы

- Целевая матрица разрешений/масштабов мониторов в залах — уточнить у
  заказчика (влияет на брейкпоинты).
- Нужен ли компакт-оверлей «поверх всех окон» в v-целевой поставке или это
  опция — от заказчика.
