# Этап 01 — Ядро захвата звука (Rust)

> Проект **«Аудиопротокол»**. Цель — рабочее ядро захвата: перечисление
> устройств, поток с одного канала через `cpal`, кольцевой буфер, сегментный
> WAV-райтер и индикаторы уровня. Это фундамент, на котором строятся этапы
> `02` (надёжность), `03` (хранилище/целостность), `04` (UI-метры).

## Контекст

Архитектура — [`../CLAUDE.md`](../CLAUDE.md). **Все числовые параметры берём из
реестра [`../docs/configuration.md`](../docs/configuration.md)** (раздел «Аудио»
и «Запись и надёжность»), а не константами. Модуль кода — `src-tauri/audio/`
(+ часть `recorder/` для сегментации). Дефолт качества — 44.1 кГц / 16 бит /
моно, WAV PCM.

## Результат этапа (Deliverables)

1. **Перечисление устройств.** Список устройств ввода + их возможности
   (поддерживаемые частоты/форматы/каналы), Tauri-команда `list_audio_devices`.
2. **Конвейер захвата.** `cpal` input stream → **кольцевой буфер** (lock-free,
   producer в аудио-callback'е, consumer в отдельном потоке) → **сегментный
   WAV-райтер** (`hound`). Параметры — из конфигурации (`sample_rate_hz`,
   `bit_depth`, `channels`, `segment_seconds`, `flush_interval_ms`).
3. **Согласование формата (нативная частота + нормализация, без ресемпла).**
   Открываем устройство в его системном/нативном формате (или в запрошенном
   `sample_rate_hz`/`channels`, если устройство его поддерживает) и пишем WAV на
   **фактической** частоте устройства. Нормализуем только дёшево: sample-формат
   (часто f32) → PCM `bit_depth` (квантование) и каналы → `channels` (downmix в
   моно). **Ресемпл частоты в v1 не делаем** — точная частота мастера некритична,
   т.к. сервер W2.11 деривирует копию `asr_derivative_hz` (16 кГц) для ASR; файлы
   остаются стандартными PCM/моно. Семантика `audio.sample_rate_hz` — «желаемая;
   если устройство не поддерживает — пишем на нативной». (Решение согласовано с
   заказчиком, отступление от первоначального ТЗ этапа.)
4. **Непрерывный сброс.** Данные пишутся на диск с интервалом `flush_interval_ms`
   (fsync), чтобы потеря при сбое не превышала этот интервал. Ротация файла —
   каждые `segment_seconds`.
5. **Индикаторы уровня.** RMS/пик по каналам, поток событий в UI с UI-частотой
   (~20–30 Гц), Tauri-event `audio_level`. Без подвисаний аудио-callback'а.
6. **Управление.** Tauri-команды `start_capture` / `stop_capture` /
   `pause_capture` / `resume_capture`; запись сегментов в `storage.root_path`.
   Заглушка манифеста (полноценный — в `03`): просто список созданных сегментов.
7. **Тесты.** Юнит: кольцевой буфер (переполнение/wrap), логика ротации
   сегментов, корректность WAV-заголовка (длина/формат/частота), приведение
   формата. Интеграция: захват с null/loopback-устройства → проверка, что
   сегменты валидны и непрерывны по времени.

## Шаги

1. Подключить `cpal`, `hound` (и при необходимости лёгкий ресемплер, напр.
   `rubato`) в `Cargo.toml`.
2. `audio/devices.rs` — перечисление и описание устройств + Tauri-команда.
3. `audio/ring.rs` — кольцевой буфер для передачи семплов из callback'а в
   consumer без блокировок и аллокаций в реальном времени.
4. `audio/capture.rs` — построение input stream, обработка нативного формата,
   приведение к целевому, отправка в кольцевой буфер; расчёт уровней.
5. `recorder/segment_writer.rs` — потребитель буфера: WAV-райтер с ротацией по
   `segment_seconds` и fsync по `flush_interval_ms`; именование сегментов
   (порядковый № + таймкод).
6. `ipc/audio_cmds.rs` — Tauri-команды и события (`audio_level`,
   `capture_state`).
7. Тесты (юнит + интеграция с null-устройством).

## Критерии приёмки (Acceptance)

- С реального микрофона ведётся запись в сегментные WAV (PCM `bit_depth`/`channels`,
  по умолчанию 16 бит / моно; частота — фактическая частота устройства); файлы
  валидны, непрерывны по времени, без пропусков/наложений на стыках сегментов.
- Индикаторы уровня двигаются и соответствуют сигналу; аудио-callback не
  блокируется (нет xrun'ов при штатной нагрузке).
- Sample-формат устройства приводится к PCM `bit_depth` (квантование) и каналы — к
  `channels` (downmix); частота пишется фактическая (без ресемпла); WAV-заголовок
  корректен и соответствует фактической частоте/каналам/разрядности.
- Параметры читаются из конфигурации; в логике нет «магических чисел».
- `cargo clippy` без warning'ов; тесты зелёные.

## Вне объёма этапа

- Восстановление после сбоя/обрыва устройства, watchdog, контроль диска,
  дублирующая дорожка — этап `02`.
- Полный манифест сессии, шифрование at-rest, SHA-256/хеш-цепочка, журнал
  событий — этап `03`.
- Выгрузка на сервер — этап `06`. Многоканал по ролям — фаза 2 (`09`).
- UI сверх минимального дёрганья команд — этап `04` (здесь достаточно ручной
  проверки через dev-команды).
