# Этап 03 — Хранилище, целостность, ретеншн

> Проект **«Аудиопротокол»**. Цель — превратить «сегменты на диске + журнал
> восстановления» (`02`) в полноценное локальное хранилище записей: SQLite-
> манифест, шифрование at-rest, SHA-256 + хеш-цепочка, журнал событий,
> экспортируемый манифест для верификации на сервере и движок локального
> ретеншна.

## Контекст

Архитектура — [`../CLAUDE.md`](../CLAUDE.md). Параметры — реестр
[`../docs/configuration.md`](../docs/configuration.md) (разделы «Хранилище и
ретеншн», «Целостность»). Код — `src-tauri/store/` + `src-tauri/integrity/`.
Опирается на `02` (журнал/сегменты).

**Границы:** привязка к делу — `05` (здесь только поле в манифесте); выгрузка
и сигнал «сервер подтвердил» — `06` (здесь — движок ретеншна + интерфейс
триггера); ГОСТ-подпись — фаза 2 (`11`).

## Ключевые проектные решения

- **SQLite — манифест записи (источник для запросов и UI).** Аварийно-
  устойчивый append-only журнал из `02` остаётся «последней инстанцией» для
  восстановления; при старте/восстановлении сверяем журнал → SQLite. Горячий
  путь (live-запись) не зависит от SQLite.
- **Хеши — по каноничному аудио-контенту (PCM/WAV), до шифрования.** Сервер
  верифицирует SHA-256 по тому же контенту, что получит при выгрузке.
  Шифрование — отдельный локальный слой at-rest, в хеш не входит.
- **Шифрование на финализации сегмента, не «на лету».** Сегмент пишется
  открытым WAV во время записи (так он crash-safe и чинится в `02`); по
  завершении сегмента — хеш по контенту → шифрование at-rest (AES-256-GCM) →
  удаление открытой копии. Половинно записанный сегмент остаётся
  восстановимым; шифрование не ломает надёжность. (ГОСТ-шифр как опция — фаза 2.)

## Результат этапа (Deliverables)

1. **SQLite-манифест** (`rusqlite`, WAL): таблицы `sessions` (метаданные,
   статус, station_id/operator_id, поле `adjudication_ref` под привязку из
   `05`, статусы выгрузки под `06`), `segments` (№, длительность, путь, размер,
   `sha256`, звено хеш-цепочки), `events` (журнал событий записи). Миграции
   схемы.
2. **Целостность.** SHA-256 каждого сегмента (`integrity.segment_hash`) +
   **хеш-цепочка** (`integrity.hash_chain`): звено = `H(prev_link || segment_hash)`,
   итог сессии — финальное звено. Считается по мере финализации сегментов.
3. **Журнал событий** (`integrity.event_log`): структурированные события
   (старт/пауза/обрыв устройства/восстановление/стоп/ротация) с метками
   времени — часть записи целостности.
4. **Шифрование at-rest** (`storage.encrypt_at_rest`): AES-256-GCM на сегмент;
   ключ — в системном защищённом хранилище ОС (keychain/secret-service/DPAPI),
   при недоступности — производный с парольной фразой станции. Прозрачное
   дешифрование при чтении (для выгрузки в `06`).
5. **Экспорт манифеста записи** (JSON): состав сегментов + хеши + цепочка +
   метаданные + события — сопровождает выгрузку, по нему сервер делает `verify`
   (контракт — `07`).
6. **Движок локального ретеншна.** По `retention.mode`
   (`until_confirmed_plus_window` дефолт): удалять локальную копию только при
   `require_integrity_verified=true` (серверное подтверждение) **и** по
   истечении `safety_window_hours`. Сам триггер «сервер подтвердил» приходит из
   `06`; здесь — политика, планировщик и безопасное удаление (файл + записи).
   Режимы `delete_on_confirm` / `manual` — поддержать.
7. **Тесты.** Корректность хеш-цепочки; детект подмены (tamper) сегмента;
   round-trip шифрования; ретеншн (не удаляет до подтверждения+окна; удаляет
   после; режимы); реконсиляция журнал→SQLite при восстановлении; схема
   экспортируемого манифеста.

## Шаги

1. `store/db.rs` — открытие SQLite (WAL), схема, миграции.
2. `store/manifest.rs` — модель сессии/сегментов/событий, запись по ходу.
3. `integrity/hash.rs` — SHA-256 сегмента + построение хеш-цепочки.
4. `integrity/events.rs` — структурированный журнал событий.
5. `store/crypto.rs` — AES-256-GCM at-rest, интеграция с keystore ОС,
   шифрование на финализации + прозрачное чтение.
6. `store/export.rs` — сериализация манифеста записи (JSON) для верификации.
7. `store/retention.rs` — движок политики + планировщик + безопасное удаление;
   интерфейс триггера подтверждения (вызовется из `06`).
8. Реконсиляция журнал→SQLite в пути восстановления (`02` `recovery.rs`).
9. Тесты.

## Критерии приёмки (Acceptance)

- Хеш-цепочка корректна; изменение любого сегмента ломает верификацию (tamper
  detection).
- Сегменты зашифрованы at-rest; открытая копия удаляется после финализации;
  чтение для выгрузки прозрачно дешифрует.
- Экспортируемый манифест достаточен для серверного `verify` (`07`).
- Ретеншн не удаляет локальную копию до серверного подтверждения целостности +
  окна безопасности; после — удаляет (файл + записи); режимы работают.
- Параметры — из конфигурации; `cargo clippy` чисто; тесты зелёные.

## Вне объёма этапа

- Привязка к делу (UI/кэш дел) — `05`; выгрузка/оффлайн-очередь и подача сигнала
  подтверждения — `06`; серверная верификация — `07`.
- ГОСТ ЭЦП (юридически значимая подпись) и ГОСТ-шифр — фаза 2 (`11`).

## Решено

- **Ключ шифрования:** keystore ОС как основной + **парольная фраза станции
  как фолбэк** (настройка) — гарантирует работу на оффлайн/headless-станции,
  где keystore ОС недоступен без активной сессии.
- **Манифест:** включаем **значимые для целостности события** (старт/стоп/
  пауза/обрыв/восстановление/ротация с метками времени); подробные
  диагностические логи остаются локально.
