# Этап 06 — Агент выгрузки в `ex_system` (оффлайн-устойчивый)

> Проект **«Аудиопротокол»**. Цель — клиентская сторона выгрузки: возобновляемая
> чанковая отправка записей в `ex_system` по контракту `07`, оффлайн-очередь,
> ретраи с бэкоффом, запрос серверной верификации и подача сигнала «сервер
> подтвердил» в движок ретеншна (`03`). Парный к серверному этапу `07`.

## Контекст

Архитектура — [`../CLAUDE.md`](../CLAUDE.md). Параметры — реестр
[`../docs/configuration.md`](../docs/configuration.md) (раздел «Выгрузка и
сеть», «Аутентификация»). Код — `src-tauri/sync/`. Источник данных — манифест
и зашифрованные сегменты (`03`); контракт эндпоинтов — `07`; статус в UI —
список сессий (`04`); триггер ретеншна — `03`.

**Приоритет — захват, не выгрузка.** Идущая запись важнее догрузки: агент не
должен мешать ядру захвата (фоновый, низкий приоритет; политика «во время
активной записи» — см. «Решено»).

## Результат этапа (Deliverables)

1. **Возобновляемая чанковая выгрузка** по контракту `07`: `sessions/` →
   `upload/init` → `part/<n>` ×N → `upload/complete` → `verify`. Размер чанка —
   `sync.chunk_size_mb`; параллельность — `sync.parallel_uploads`. Чтение
   сегментов из хранилища с прозрачным дешифрованием (`03`); хеши — из
   манифеста.
2. **Идемпотентность и докачка.** Трекинг отправленных частей в манифесте;
   после обрыва — продолжение с неотправленных; повтор `part/<n>` безопасен.
3. **Оффлайн-очередь.** Персистентная очередь записей/частей на выгрузку,
   переживающая рестарт приложения; обработка при появлении сети и валидного
   токена.
4. **Ретраи/бэкофф.** Экспоненциальный бэкофф (`retry.backoff_base_ms` →
   `retry.backoff_max_ms`), лимит `retry.max_attempts` (0 = до успеха);
   различать временные (сеть/5xx) и постоянные (4xx) ошибки.
5. **Верификация и сигнал ретеншна.** После `complete` — запрос `verify`; при
   `integrity_verified=true` — отметка в манифесте и **вызов триггера ретеншна
   `03`** (это и есть условие удаления локальной копии по
   `until_confirmed_plus_window`). При несовпадении — статус «ошибка
   целостности», локальную копию **не удалять**.
6. **Аутентификация.** Выгрузка под JWT (станция + оператор, `07`); при
   отсутствии/протухании токена — копить в очереди (не терять данные),
   рефреш/повторный вход по возможности. Идущую запись это не затрагивает.
7. **Статусы в UI.** Прогресс и состояние (в очереди / выгружается N% /
   выгружено / проверено / ошибка) — в список сессий (`04`), по данным
   манифеста; повтор/пауза выгрузки оператором.
8. **Тесты.** Возобновление после обрыва (докачка с середины); персистентность
   очереди через рестарт; бэкофф и классификация ошибок; `verify=true` →
   триггер ретеншна, `verify=false` → копия сохранена; поведение без токена
   (накопление, не потеря); идемпотентность `part`.

## Шаги

1. `sync/client.rs` — HTTP-клиент (`reqwest`) под контракт `07`, JWT.
2. `sync/queue.rs` — персистентная оффлайн-очередь (в манифесте/SQLite `03`).
3. `sync/uploader.rs` — чанковая логика init/part/complete + докачка +
   параллельность + бэкофф.
4. `sync/verify.rs` — запрос `verify`, разбор результата, вызов триггера
   ретеншна `03`.
5. `sync/scheduler.rs` — обработка очереди по сети/токену/приоритету (захват
   важнее).
6. Проброс статусов в UI (`04`) и тесты.

## Критерии приёмки (Acceptance)

- Запись полностью выгружается и верифицируется; при обрыве сети выгрузка
  продолжается с неотправленных частей после восстановления связи.
- Очередь переживает рестарт приложения; без сети/токена данные копятся, не
  теряются.
- `verify=true` запускает ретеншн (`03`), `verify=false` оставляет локальную
  копию + статус ошибки целостности.
- Выгрузка не мешает идущей записи (приоритет захвата соблюдён).
- Параметры — из конфигурации; `cargo clippy` чисто; тесты зелёные.

## Вне объёма этапа

- Серверные эндпоинты приёма/верификации/связывания — `07`.
- Сама политика удаления и шифрование — `03` (здесь только триггер и чтение).
- ГОСТ-подпись при выгрузке — фаза 2 (`11`).

## Решено

- **Момент выгрузки:** авто-фоном по готовности записи (`sync.auto_upload`),
  оператор может ставить на паузу/повторять вручную.
- **Во время активной записи:** фоновая выгрузка низким приоритетом, не в ущерб
  захвату; «отложить тяжёлую догрузку до конца сессии» — настройка
  `sync.defer_during_recording` (дефолт `false`).
