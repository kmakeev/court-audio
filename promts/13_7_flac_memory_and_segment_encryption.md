# Этап 13.7 — R-013 (шифрование сегментов в live-пути) + R-014 (потоковый FLAC-экспорт)

> Проект **«Аудиопротокол»**, два пункта бэклога
> ([`../docs/backlog.md`](../docs/backlog.md)), найденные анализом
> ресурсоёмкости/кода 2026-07-05: **R-013** (безопасность — заявленное
> документацией шифрование аудио-сегментов at-rest фактически не подключено в
> путь записи) и **R-014** (надёжность — FLAC-экспорт держит всю сессию в
> памяти, риск OOM на длинных заседаниях). Пункты независимы; собраны в один
> проход, но правятся и принимаются раздельно.
>
> **Порядок важен.** R-013 меняет формат файлов на диске (`.wav` → `.enc`) —
> сначала прогнать полный цикл «запись → рестарт/восстановление → прослушивание
> → экспорт → зеркало → выгрузка → ретеншн» на dev-станции, потом принимать.
> R-014 — локализуемая правка `export/flac.rs`.

## Контекст

Криптодвижок готов и ждёт подключения:
[`store::crypto::finalize_segment`](../src-tauri/src/store/crypto.rs) — SHA-256
по открытому WAV → AES-256-GCM в `<seg>.wav.enc` → удаление открытой копии;
потребители уже прозрачны к `.enc` (`read_segment_plain` в
[`sync::uploader`](../src-tauri/src/sync/uploader.rs), плеер и экспорт резолвят
ключ станции). Ключ станции обязателен и валидируется на старте (R-004, этап
13.5). Экспортная склейка потокова
([`export::audio::IntSegmentCursor`](../src-tauri/src/export/audio.rs)), но
FLAC-ветка ([`export::flac`](../src-tauri/src/export/flac.rs)) читает весь WAV
в `Vec<i32>` и сериализует поток в память.

## Часть A — R-013: подключить шифрование сегментов в путь записи

### Суть

При `storage.encrypt_at_rest = true` (дефолт) закрытый сегмент должен
финализироваться (`finalize_segment`) сразу после закрытия файла райтером:
на диске остаётся только `.enc`, открытый WAV удаляется. Сейчас вызовов
`finalize_segment` из боевого пути нет — сегменты лежат plaintext'ом, что
расходится с README/`configuration.md`/`CLAUDE.md` («шифрование записей
at-rest»).

### Объём

1. **Точка вызова** — поток райтера (`audio::capture::run_consumer` /
   обработка `drain_completed`), **вне** аудио-callback'а: шифрование целого
   сегмента (~2.6 МБ при дефолтах) на writer-потоке не мешает захвату
   (кольцевой буфер даёт 2 с запаса). Хеш из `FinalizedSegment` — тот же, что
   сейчас считает реконсиляция (по **открытому** содержимому; хеш-цепочка не
   меняется).
2. **Журнал/манифест** — фиксировать фактический `stored_path` (`.enc`) и
   признак шифрования; реконсиляция (`store::reconcile`) хеширует через
   `read_segment_plain` (а не по сырому файлу), уже финализированные сегменты
   не перечитывает повторно без нужды.
3. **Восстановление после сбоя** — незавершённый «хвостовой» сегмент лежит
   открытым WAV (усечённый, читаемый — принцип этапа 02). Реконсиляция при
   старте дофинализирует его: хеш → шифрование → `.enc`.
4. **Зеркало** (R-010) — копирует файлы «как есть», значит после подключения
   на зеркало едет `.enc`; метаданные зеркала не меняются. Проверить копирование
   по факту закрытия (момент: до или после финализации — выбрать «после», чтобы
   зеркало не хранило plaintext дольше основного места).
5. **Ретеншн** (`store::retention`) — удаляет по `stored_path` из манифеста.
6. **Fail-secure** — при `encrypt_at_rest = true` и недоступном ключе запись
   должна завершаться **громкой ошибкой** (баннер + журнал), не тихим
   plaintext-фолбэком (продолжение линии R-003/R-004). Self-test «Ключ станции»
   уже даёт Fail — сверить формулировки.
7. **Выключенный флаг** (`encrypt_at_rest = false`) — поведение ровно текущее
   (plaintext WAV), без `.enc`; смешанные сессии (флаг переключили между
   сессиями) читаются прозрачно (`read_segment_plain` уже умеет оба вида).

### Критерии приёмки (A)

- [ ] Запись при дефолтах: после стопа в каталоге сессии **нет** открытых
      `.wav`, только `.enc`; плеер, экспорт (WAV и FLAC), выгрузка в
      `ex_system` и HTML-плеер экспорта работают без изменений.
- [ ] «Питание выдернули» посреди записи (kill процесса): при рестарте сессия
      восстанавливается, хвостовой сегмент дофинализирован в `.enc`, хеш-цепочка
      сходится (`verify_chain`).
- [ ] Зеркало содержит те же `.enc` + метаданные; независимая реконсиляция по
      зеркалу проходит.
- [ ] Без ключа станции при `encrypt_at_rest = true` старт записи невозможен с
      понятной ошибкой (не «тихий plaintext»).
- [ ] `encrypt_at_rest = false` — байт-в-байт текущее поведение.
- [ ] Выгрузка ранее записанных **plaintext**-сессий (сделанных до этапа) идёт
      как раньше — миграция старых сессий не требуется.

## Часть B — R-014: потоковый FLAC-экспорт

### Суть

`encode_wav_to_flac` не должен держать в памяти всю сессию: цель — пиковая
память **O(блок/сегмент)**, не O(сессия). Трёхчасовая моно-сессия сейчас
требует ≈1.9 ГБ RAM (`Vec<i32>` + `ByteSink`) — риск OOM на станции с 4 ГБ.

### Объём

1. Заменить «весь WAV → `MemSource` → `ByteSink` → файл» на потоковое
   кодирование: собственный `flacenc::source::Source` поверх посегментного
   чтения (образец курсора — `IntSegmentCursor`) либо кодирование
   фиксированными блоками с дозаписью в файл через `io::Write`-sink.
2. Семантика прежняя: `encode_with_fixed_block_size`, «хвост» дополняется
   тишиной внутри последнего блока, `STREAMINFO.total_samples` — настоящая
   длина (см. doc-комментарий модуля).
3. Прогресс-события экспорта (если появятся паузы >1 с на длинных сессиях) —
   переиспользовать существующий канал прогресса мастера, новых параметров
   реестра не вводить.

### Критерии приёмки (B)

- [ ] Roundtrip-тест (`claxon`) зелёный: декодированные семплы и длина
      совпадают с входом, включая «хвостовой» неполный блок.
- [ ] Экспорт длинной сессии (сгенерировать ≥1 ч синтетики) в FLAC: пиковая
      память процесса — сотни МБ, а не гигабайты (замер `time -l` /
      Task Manager на dev-станции; ориентир — < 300 МБ поверх базовой).
- [ ] Байтовая идентичность с текущим энкодером **не требуется** (другой
      порядок буферизации допустим); валидность потока — по декодеру.

## Инварианты (обе части)

- Аудио-callback и кольцевой буфер не трогаем: никакой криптографии/кодирования
  в realtime-пути.
- Никаких новых параметров реестра без нужды; поведение управляется
  существующими `storage.encrypt_at_rest` / `export.default_codec`.
- Язык кода/доков — русский; секреты в `settings.json` не появляются.
