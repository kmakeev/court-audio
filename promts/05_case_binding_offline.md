# Этап 05 — Привязка записи к делу (оффлайн)

> Проект **«Аудиопротокол»**. Цель — привязать запись к делу так, чтобы это
> работало **без сети в зале**: кэш списка дел из `ex_system` (синхронизируется
> при наличии связи) + ручной ввод № как фолбэк. Результат привязки уходит в
> манифест и далее на сервер (`06`/`07`) для связывания с `Adjudication`.

## Контекст

Архитектура — [`../CLAUDE.md`](../CLAUDE.md). Параметры кэша — реестр
[`../docs/configuration.md`](../docs/configuration.md), раздел «Привязка к делу
(кэш дел на станции)». Код — `src-tauri/store/` (кэш) + `src/` (UI привязки на
экране «Запись» из `04`). Поле `adjudication_ref` в манифесте — из `03`.

**ПДн:** кэш содержит № дел и ФИО — чувствительно. Кэш **шифруется at-rest**
(`case_cache.encrypt`), минимальный состав полей, лимит `max_records`,
устаревание по `ttl_hours`. На сервере список — **докет суда/зала** (а не
личные дела оператора), отдаётся через slim-эндпоинт со station-доступом
(см. `07`).

## Модель привязки

Привязка имеет два состояния:

- **Resolved** — выбрано дело из кэша → в манифест пишется `adjudication_id`.
- **Pending (manual)** — оператор ввёл № / ФИО вручную (нет в кэше или
  оффлайн без кэша) → в манифест пишется сырой идентификатор; финальное
  связывание с `Adjudication` делает сервер/оператор позже (при выгрузке `06`,
  привязке на сервере `07`, либо подтверждении в W2.11).

Запись **никогда не блокируется** отсутствием привязки: можно начать без неё и
привязать потом (но в UI — заметное напоминание).

## Результат этапа (Deliverables)

1. **Синхронизация кэша дел.** Tauri-команда/фоновая задача: при наличии сети и
   валидной сессии оператора — тянуть **докет суда/зала** из `ex_system` (slim-
   эндпоинт списка дел для станции, см. `07`), сохранять в **зашифрованный**
   локальный кэш (минимум полей: `id`, № дела, ФИО подсудимых, дата).
   Лимит/скоуп/ttl — из конфигурации.
2. **Оффлайн-чтение и поиск.** Поиск по кэшу (№ дела/ФИО) на экране «Запись»;
   работает без сети. Индикатор свежести кэша (когда синхронизировано;
   «устарел» по `ttl_hours`).
3. **Ручной ввод (фолбэк).** Если дела нет в кэше/нет кэша — ввод № (и опц.
   ФИО) вручную → привязка `pending`.
4. **Запись привязки в манифест.** `adjudication_ref` = `{kind: resolved|manual,
   adjudication_id?, raw_number?, raw_fio?}`; смена/уточнение привязки уже
   существующей записи.
5. **UI на экране «Запись» (из `04`).** Превратить «поле-заглушку» привязки в
   реальный пикер: автокомплит по кэшу + переключатель «ввести вручную» +
   индикатор статуса привязки (resolved/pending) + свежесть кэша.
6. **Серверная сторона (в `ex_system`, с `07`).** Slim-эндпоинт списка дел для
   станции в скоупе **докета суда/зала** (только нужные поля, без тяжёлых
   вложений) + permission «станция/оператор читает докет своего зала».
   User-scoped `/adjudications/` недостаточен — он отдаёт лишь личные дела.
7. **Тесты.** Кэш: шифрованный round-trip, лимит/скоуп/ttl, поиск; модель
   привязки (resolved/manual); запись в манифест; оффлайн-сценарий (нет сети →
   ручной ввод → pending); UI-тесты пикера (vitest).

## Шаги

1. `store/case_cache.rs` — зашифрованный кэш дел, sync-команда, поиск, ttl.
2. Расширить манифест (`03`) полем `adjudication_ref` и операциями привязки.
3. `src/` — пикер дел на экране «Запись» (автокомплит + ручной ввод + статусы).
4. Сверка/уточнение полей списка в `ex_system` (с `07`); при необходимости —
   slim-эндпоинт.
5. Тесты (Rust + vitest).

## Критерии приёмки (Acceptance)

- При наличии сети кэш дел синхронизируется; в оффлайне привязка по кэшу
  работает; индикатор свежести корректен.
- Если дела нет — ручной ввод даёт привязку `pending`, запись не блокируется.
- Привязка пишется в манифест в обоих состояниях; уходит на сервер (`06`/`07`)
  достаточной для связывания с `Adjudication`.
- Кэш зашифрован at-rest, ограничен скоупом/лимитом/ttl (ПДн-минимизация).
- Параметры — из конфигурации; тесты зелёные; `tsc`/`cargo clippy` чисто.

## Вне объёма этапа

- Сама выгрузка записи и оффлайн-очередь — `06`; серверное связывание с
  `Adjudication` и slim-эндпоинт «по-боевому» — `07`.
- Создание нового дела со станции (только привязка к существующему) — не в
  объёме v1.

## Решено

- **Скоуп кэша:** докет суда/зала (`court_docket`) — станция привязывает любую
  запись к делу, назначенному в её зале. Требует серверного slim-эндпоинта со
  station-доступом (см. `07`), не user-scoped `/adjudications/`.
- **Обновление кэша:** при старте + вручную + по `ttl_hours` (24 ч), без
  постоянного фонового опроса.
