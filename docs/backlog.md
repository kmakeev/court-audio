## Предложения (B)

### B-001. Полностью автономный (офлайн) запуск станции по PIN

- **Статус:** ✅ принято (→ [`promts/13_6_offline_pin_and_overlay.md`](../promts/13_6_offline_pin_and_overlay.md); реализация — **после согласования** открытых вопросов) · **Дата:** 2026-07-03 · **Автор:** заказчик/оператор
- **Контекст.** Сейчас офлайн-старт возможен **только после** хотя бы одного
  онлайн-входа оператора в пределах окна `auth.operator.cached_session_hours`
  (дефолт 24 ч): вход кэширует сессию + хеш PIN, дальше зал работает автономно
  (`auth_unlock_offline` по PIN). См. [`auth.md`](auth.md),
  `src-tauri/src/ipc/auth_cmds.rs` (`OfflineUnlockError::Expired`). В изолированном
  зале, где backend `ex_system` **вообще недоступен** (нет сети/API), станцию
  **нельзя запустить впервые** — нет ни кэша сессии, ни возможности войти онлайн.
- **Предложение.** Автономный режим первого запуска **без обязательного
  онлайн-входа**: разблокировка по **PIN, заданному при установке/провижининге
  или в конфигурации** станции; идентичность оператора/станции — из
  провижининг-профиля, а не из онлайн-сессии. Выгрузка (когда связь появится) —
  через отложенную очередь, как сейчас.
- **Открытые вопросы.**
  - Как получить `operator_id`/`station_id` без сервера — провижининг-файл на
    станции? предзаданный маппинг «PIN → оператор»? один «сервисный оператор
    зала»?
  - Хранение секрета: PIN — только **хеш** (Argon2id), как в кэш-сессии; в
    `settings.json` секреты не хранятся (текущее правило).
  - Согласование с решением заказчика «оператор **обязан** авторизоваться»
    (`CLAUDE.md`, `auth.md`): автономный режим меняет модель идентичности —
    требует явного согласования.
  - Как `ex_system` примет записи с «офлайн-провиженной» идентичностью
    (валидация `operator_id`/`station_id` при догрузке; контракт `07`).
  - Докет дел офлайн: предзагруженный/импортируемый `case_cache`, если онлайн-
    синхронизации не было ни разу.
- **Готовый механизм (переиспользовать).** Для **администратора** такой офлайн-
  провижининг **уже реализован**: PIN задаётся при развёртывании через env
  `COURT_AUDIO_ADMIN_PIN` и хранится как Argon2id-хеш в зашифрованном блобе
  `admin_pin.enc` (AES-256-GCM ключом станции), засев при старте — идемпотентно,
  без онлайн-входа (`src-tauri/src/store/admin_pin.rs`,
  `provision_from_env_if_absent` в `lib.rs`; политика — `admin.pin` в
  `settings.json`). B-001 по сути = **распространить этот же паттерн на
  операторскую идентичность**: провижининг `operator_id`/`station_id` + PIN
  (env/файл провижининга) → зашифрованный блоб, разблокировка офлайн как
  `auth_unlock_offline`. Отличие от админ-PIN — оператору нужна ещё **привязка к
  `operator_id`/`station_id`** (админ-PIN идентичность не несёт).
- **Затрагивает.** `sync/auth` (провижининг vs cached-session), `store::admin_pin`
  (образец), `SessionMeta` (контракт `07`), `configuration.md` (новые ключи
  провижининга), [`auth.md`](auth.md), приём в `ex_system` (валидация
  идентичности), провижининг/установка ([`packaging.md`](packaging.md)).

## Замечания и технический долг (R)

### R-001. Неподписанные сборки macOS/Windows блокируются ОС

- **Статус:** 🆕 новое · **Дата:** 2026-07-03 · **Автор:** проверка v0.2.0
- **Контекст.** Релизный `release.yml` собирает macOS (`.dmg`) и Windows (`.exe`)
  **без подписи/нотаризации**. macOS на Apple Silicon отдаёт «приложение
  повреждено» (Gatekeeper + карантин), Windows — предупреждение SmartScreen.
  Обходной приём и правильное решение задокументированы в
  [`packaging.md`](packaging.md) («Подпись пакетов» → macOS/Windows).
- **Суть.** Для боевой раздачи подключить подпись в CI: macOS — Developer ID +
  нотаризация; Windows — Authenticode. Секреты — в контуре. Это объём этапа
  `08_packaging` (ГОСТ-подпись — фаза 2, `11`).
- **Затрагивает.** `release.yml`, [`packaging.md`](packaging.md).

### R-002. Сегмент больше `chunk_size_mb` роняет выгрузку без дружелюбной реакции

- **Статус:** 🆕 новое · **Дата:** 2026-07-03 · **Автор:** проверка выгрузки
- **Контекст.** Часть выгрузки = целый сегмент. Если размер сегмента больше
  `sync.chunk_size_mb`, клиент валит выгрузку с `SyncError::Permanent` («часть N
  больше предела…», статус `Failed`, **до сети** — в логах backend пусто).
  Реальный кейс: `recorder.segment_seconds=300` при 44.1 кГц/16 бит/моно ≈ 25 МиБ
  против дефолта `chunk_size_mb=8`. Ограничение и формула добавлены в
  [`configuration.md`](configuration.md) («Размер части ↔ размер сегмента»).
- **Предложение.** Убрать «немой» отказ: (а) валидация настроек при старте —
  предупреждать, если `segment_size > chunk_size_mb` (или `> серверного лимита`);
  (б) опц. авто-подстройка `chunk_size_mb`/деление сегмента; (в) согласовать
  дефолты `recorder.segment_seconds` и `sync.chunk_size_mb`/`AUDIO_MAX_UPLOAD_PART_MB`,
  чтобы «из коробки» не падало.
- **Затрагивает.** `sync/uploader.rs`, валидация настроек, `configuration.md`,
  серверный `AUDIO_MAX_UPLOAD_PART_MB` (`ex_system`).

### R-003. 🔴 Невалидный `settings.json` обходит гейты PIN/авторизации (fail-open)

- **Статус:** ✅ принято (→ [`promts/13_5_gates_failsecure.md`](../promts/13_5_gates_failsecure.md)) (**безопасность**) · **Дата:** 2026-07-03 · **Автор:** проверка на станции
- **Наблюдение.** Если испортить `settings.json`, сделав его **невалидным**
  (битый JSON), приложение **стартует без запроса админ-PIN и авторизации
  оператора**. То есть при ошибке разбора настроек гейты доступа
  **размыкаются** (fail-open), а не смыкаются (fail-secure).
- **Риск.** Обход операторской авторизации и админ-PIN простой порчей
  конфиг-файла (локальный доступ к файлу → обход разграничения доступа этапа
  10.4). Чувствительно для контура с ПДн: политики `admin.pin.required` /
  `auth.operator.required_to_start` перестают действовать.
- **Ожидаемое поведение (fail-secure).** При ошибке чтения/разбора
  `settings.json`: **не ослаблять** гейты — применять политики как включённые
  (эквивалент `Settings::default()`, где `required = true`), и/или блокировать
  старт с явной ошибкой «конфигурация повреждена — обратитесь к администратору»,
  а не пускать дальше в открытом режиме.
- **Где смотреть.** `load_settings`/`get_settings`
  ([`ipc/mod.rs`](../src-tauri/src/ipc/mod.rs)) — сейчас при parse-ошибке
  возвращается `Err` («повреждённый файл не валит приложение»); проверить, как на
  это реагируют гейт старта (`ensure_start_allowed`, `ipc/auth_cmds.rs`) и
  админ-гейт (`ipc/admin_cmds.rs`): вероятно, ошибка проглатывается на уровне UI
  и политики просто не применяются.
- **Затрагивает.** `ipc/mod.rs` (`load_settings`), `ipc/auth_cmds.rs`
  (`ensure_start_allowed`), `ipc/admin_cmds.rs` (гейт), `settings.rs`,
  [`auth.md`](auth.md) (модель гейтов).

### R-004. Операторский PIN/офлайн-сессия не сохраняется (нет ключа станции)

- **Статус:** ✅ принято (→ [`promts/13_5_gates_failsecure.md`](../promts/13_5_gates_failsecure.md)) · **Дата:** 2026-07-03 · **Автор:** проверка на станции (Windows)
- **Наблюдение.** На Windows оператор входит онлайн по имени/паролю, задаёт PIN,
  вход проходит — но **файл с хешем PIN/кэшем сессии не создаётся**. В следующий
  раз офлайн-вход по PIN недоступен, возможен только онлайн-вход по имени/паролю.
- **Вероятная причина.** Персист офлайн-кэша идёт зашифрованно ключом станции
  (`store::auth_cache` → `crypto::resolve_station_key`). При дефолтном
  `storage.key_source = os_keystore` провайдер — **заглушка**
  (`OsKeystoreKeyProvider` → `KeystoreUnavailable`, интеграция keystore — этап
  `08`), фолбэк — парольная фраза из env `COURT_AUDIO_STATION_PASSPHRASE`
  ([`store/crypto.rs`](../src-tauri/src/store/crypto.rs) `resolve_station_key`).
  Если фраза не задана → `PassphraseMissing`, зашифрованный блоб **не пишется**.
  На станции Windows переменная, судя по всему, не задана — ключа нет.
- **Масштаб (важно).** Тот же ключ нужен **всем** at-rest блобам: `admin_pin.enc`
  (R/связь с админ-PIN), `case_cache.enc` (докет офлайн), шифрование сегментов
  (`storage.encrypt_at_rest`). Без ключа они тихо не создаются/деградируют — то
  есть без провижининга ключа станции офлайн-контур и шифрование ПДн **не
  работают целиком**, а не только PIN.
- **Ошибка проглатывается.** Онлайн-вход при этом «успешен» — сбой записи кэша,
  видимо, не доводится до оператора (нет явного предупреждения «офлайн-режим
  недоступен: не задан ключ станции»). Пересекается с fail-open поведением
  (см. R-003).
- **Ожидаемо.** (а) Провижининг ключа станции сделать **обязательным и явным**
  при развёртывании (env `COURT_AUDIO_STATION_PASSPHRASE` или реальная интеграция
  OS-keystore — этап `08`), с валидацией на старте; (б) при невозможности
  зашифровать — **явно уведомлять** оператора/админа, а не молча продолжать;
  (в) задокументировать в [`packaging.md`](packaging.md)/[`configuration.md`](configuration.md)
  провижининг фразы рядом с `COURT_AUDIO_ADMIN_PIN`.
- **Обходной приём (✅ подтверждён 2026-07-03 на станции Windows).** Задать
  `COURT_AUDIO_STATION_PASSPHRASE` до запуска приложения (как
  `COURT_AUDIO_ADMIN_PIN`): ключ выводится (Argon2id + соль `key.salt`), блобы
  (PIN оператора, сессия, `admin_pin.enc`) создаются, офлайн-вход по PIN и
  админ-PIN работают. Подтверждает корневую причину — отсутствие ключа станции.
  Остаётся сделать провижининг ключа **обязательным/явным** и не проглатывать
  сбой шифрования (см. «Ожидаемо»).
- **Затрагивает.** `store/crypto.rs` (`resolve_station_key`, `OsKeystoreKeyProvider`,
  `PassphraseKeyProvider::from_env`), `store/auth_cache.rs`, `store/admin_pin.rs`,
  `ipc/auth_cmds.rs` (реакция на сбой записи кэша), [`auth.md`](auth.md),
  [`packaging.md`](packaging.md)/[`configuration.md`](configuration.md), этап `08`
  (keystore); связь с [B-001] и R-003.

### R-005. Отсутствующий `audio.device` даёт молчаливую тишину; конфиг не переносим между ОС

- **Статус:** ✅ принято (→ [`promts/13_1_ui_fixes.md`](../promts/13_1_ui_fixes.md)) · **Дата:** 2026-07-04 · **Автор:** проверка на станции (перенос конфига macOS→Windows)
- **Наблюдение.** Перенос `settings.json` с macOS на Windows притащил
  `audio.device = "Микрофон MacBook Air"` — устройства с таким именем на Windows
  нет. Итог: список устройств заполнен, но индикатор уровня **всегда тишина**,
  **без явной ошибки**. После выбора верного устройства в настройках — ОК.
- **Дефект робастности.** Когда сохранённый `audio.device` не совпадает ни с
  одним присутствующим устройством, живой индикатор/захват **молча деградируют в
  тишину** вместо явного «выбранный микрофон не найден — проверьте настройки»
  (самотест `selected_present` в `ipc/selftest_cmds.rs` такой случай ловит, но до
  экрана «Запись»/индикатора это не доводится; `pick_device` на записи вернул бы
  `DeviceNotFound`, но пользователь видит тишину раньше).
- **Непереносимость конфига.** `settings.json` **машинно-зависим**: `audio.device`
  (и потенциально пути `storage.root_path`/`reliability.mirror.path`) не переносятся
  между ОС/машинами.
- **Ожидаемо.** (а) Предупреждать в UI экрана «Запись»/самотесте, если выбранное
  устройство отсутствует; опц. авто-фолбэк на устройство по умолчанию с
  уведомлением. (б) Задокументировать в [`configuration.md`](configuration.md)/
  [`packaging.md`](packaging.md): при переносе конфига очищать `audio.device` и
  машинно-зависимые пути.
- **Затрагивает.** `audio/capture.rs` (`pick_device`), `ipc/selftest_cmds.rs`
  (`selected_present` → довести до индикатора), UI «Запись»/«Настройки»,
  [`configuration.md`](configuration.md)/[`packaging.md`](packaging.md).

### R-006. Windows: «окно поверх» (compact overlay) — белая рамка и подвисание старт/стоп

- **Статус:** ✅ принято (→ [`promts/13_6_offline_pin_and_overlay.md`](../promts/13_6_offline_pin_and_overlay.md)) · **Дата:** 2026-07-04 · **Автор:** проверка на станции (Windows)
- **Наблюдение.** Под Windows при активации режима «окно поверх» (компактный
  overlay) появляется **белая рамка**, и приложение **перестаёт реагировать на
  старт/стоп записи** (управление подвисает).
- **Где смотреть.** `ipc/ui_cmds.rs::open_compact_overlay` создаёт окно
  `WebviewWindowBuilder` (label `overlay`, URL `index.html?window=overlay`,
  `always_on_top(true)`, `decorations(false)`). Вероятные причины (проверить на
  Windows/WebView2): (а) **белая рамка** — WebView2 показывает белый фон до
  первого рендера, либо маршрут `?window=overlay` не отрисовывается в отдельном
  окне на Windows, либо нужен `transparent`/фон; (б) **подвисание старт/стоп** —
  overlay как отдельное окно перехватывает фокус/ввод (`always_on_top`), либо
  команды записи завязаны на состояние/окно, недоступное из overlay, либо
  создание окна блокирует поток UI.
- **Ожидаемо.** Overlay корректно рендерится на Windows (без белой рамки), а
  управление записью остаётся отзывчивым из основного окна и/или overlay.
- **Затрагивает.** `ipc/ui_cmds.rs` (`open_compact_overlay`/`close_compact_overlay`),
  фронт-маршрут окна `?window=overlay`, WebView2 (Windows), `ui.compact_overlay`/
  `ui.hall_mode` (этап 10.5/10.6), экран «Запись».

### R-007. Настройки → «Разметка»: не вводится запятая, курсор прыгает в конец

- **Статус:** ✅ принято (→ [`promts/13_1_ui_fixes.md`](../promts/13_1_ui_fixes.md)) · **Дата:** 2026-07-04 · **Автор:** проверка на станции
- **Наблюдение.** В «Настройки → Разметка: метки и роли», поля «Категории
  закладок» и «Роли говорящих» — **не вводится запятая**, а **курсор прыгает в
  конец** строки при вводе. Та же болезнь у поля «Скорости (через запятую)» в
  разделе плеера.
- **Причина (найдена).** Контролируемый input, у которого
  `value = array.join(', ')`, а `onChange` **на каждый keystroke** пересобирает
  массив через `splitList` (`split(',')` + `trim` + отброс пустых) —
  `src/screens/Settings.tsx:138-147` (категории/роли), `:172-176`
  (`playback_rates`); `splitList`/`splitNumberList` в
  `src/screens/settings-common.tsx:88`. Каждый ввод переписывает всё значение →
  React сбрасывает каретку в конец; хвостовая запятая создаёт пустой сегмент,
  который `filter` отбрасывает → запятую «не набрать»; `trim` ломает ввод
  пробелов в середине.
- **Ожидаемо.** Редактировать **сырой текст** в локальном стейте поля; парсить
  (`splitList`) только на `blur`/сохранении, не реформатируя `value` на каждый
  keystroke. Буфер редактирования отделить от разобранного массива. Починка —
  единым паттерном для всех полей «через запятую».
- **Затрагивает.** `src/screens/Settings.tsx` (поля-списки категорий/ролей/
  скоростей), `src/screens/settings-common.tsx` (`splitList`/`splitNumberList`).

### R-008. Многоканал: мониторинг уровня до записи покрывает только дорожку 0

- **Статус:** ✅ принято (→ [`promts/13_2_multichannel_monitor.md`](../promts/13_2_multichannel_monitor.md)) · **Дата:** 2026-07-04 · **Автор:** проверка на станции (многоканал)
- **Наблюдение.** В блоке «Источник звука» индикаторы дорожек **до старта** и **во
  время** записи ведут себя по-разному. При записи все дорожки показывают живой
  уровень — в т.ч. когда **один и тот же источник** указан в «Администрировании»
  для нескольких дорожек (дубли микрофона активны). До записи индикаторы
  «повторяющихся» микрофонов **молчат** — активность показывает лишь одна дорожка.
- **Причина (найдена).** Живой мониторинг до записи — это **одиночный** поток
  `MonitorSession` по единственному устройству `settings.audio.device` с жёстким
  `track_id: 0` (`build_params` в [`ipc/audio_cmds.rs:664-678`](../src-tauri/src/ipc/audio_cmds.rs#L664),
  `start_monitor` → `MonitorSession::start` в
  [`audio/capture.rs:437`](../src-tauri/src/audio/capture.rs#L437) — один `cpal`-поток,
  один `run_monitor`). Значит монитор эмитит события **только под `track_id 0`**, а
  `TrackMeters` (`src/screens/Record.tsx:977`) рисует метр на **каждую**
  сконфигурированную дорожку — все дорожки, кроме 0-й, остаются тихими. При записи
  же поднимается многодорожечная ветка (`MultiTrackSession`), где **каждая** дорожка
  открывает своё устройство и эмитит собственный `track_id`
  ([`ipc/audio_cmds.rs:315-322`](../src-tauri/src/ipc/audio_cmds.rs#L315)) — отсюда и
  расхождение (при записи «правильнее»).
- **Ожидаемо.** Мониторинг в многоканале должен покрывать **все** дорожки (поднять
  превью-поток на каждую дорожку/устройство карты дорожек, эмитить уровни под их
  `track_id`), чтобы картинка «до» совпадала с «во время». Иначе оператор не может
  до старта проверить, что дубли микрофона реально пишут сигнал.
- **Затрагивает.** `ipc/audio_cmds.rs` (`start_monitor`/`build_params` — многоканальный
  монитор), `audio/capture.rs` (`MonitorSession` — вероятно нужна много-дорожечная
  версия), `src/screens/Record.tsx` (`TrackMeters` — уже готов к N дорожкам).

### R-010. Зеркало дублирующей дорожки: плоская свалка аудио без структуры и метаданных

- **Статус:** ✅ принято (→ [`promts/13_3_mirror_structure.md`](../promts/13_3_mirror_structure.md)) · **Дата:** 2026-07-04 · **Автор:** проверка на станции (зеркало)
- **Наблюдение.** При `reliability.mirror.enabled=true` с заданной папкой в неё
  попадают **только аудиофайлы**, причём сегменты **всех сессий и всех дорожек — в
  один каталог** и **без метаданных**. Восстановить по зеркалу, к какой сессии/делу/
  дорожке относится файл, невозможно.
- **Причина (найдена).** `Mirror::mirror_segment` копирует сегмент **по имени файла в
  единый `dir`** (`reliability.mirror.path`) — без подкаталогов сессии/дорожки
  ([`reliability/mirror.rs:31-37`](../src-tauri/src/reliability/mirror.rs#L31)). Каталог
  зеркала статичен из настроек и **не зависит от сессии**
  ([`audio/capture.rs:288-297`](../src-tauri/src/audio/capture.rs#L288)). Для сравнения,
  основное хранилище держит **структуру**: каталог сессии → подкаталоги дорожек
  (`multitrack::track_dir`), `tracks.json` (`write_track_map`), журнал-манифест
  (`journal`), SQLite-манифест и интегрити (SHA-256/хеш-цепочка). Ничего из этого в
  зеркало не переносится.
- **Риск коллизий.** Имена сегментов — `seg-{index:04}-{unix_ms}.wav`
  ([`segment_writer.rs:240`](../src-tauri/src/recorder/segment_writer.rs#L240)), индекс
  стартует с 1 **на каждую дорожку**. При сбросе разных дорожек/сессий в один плоский
  каталог совпадение имён вероятно, а `std::fs::copy` **тихо перезапишет** файл —
  потенциальная потеря данных зеркала (при том что зеркало задумано как страховка).
- **Ожидаемо (изучить хранение и дальнейшую работу).** Зеркало должно **повторять
  структуру основного места**: каталог сессии → подкаталоги дорожек, плюс копия
  метаданных (`tracks.json`, журнал/манифест, интегрити), чтобы зеркало было
  **самодостаточным** для восстановления и последующей выгрузки/приёмки в `ex_system`
  (иначе «второй носитель» не заменяет основной при отказе — нечем связать записи с
  делом). Решить: зеркалить **посегментно с реконструкцией дерева** или **синхронизировать
  каталог сессии целиком** (rsync-подобно, включая метаданные) по завершении.
- **Затрагивает.** `reliability/mirror.rs` (`Mirror` — структура пути + метаданные),
  `audio/capture.rs` (`mirror_dir` → путь на сессию/дорожку), `recorder/multitrack.rs`
  (`track_dir`/`tracks.json` — что копировать), `store/`+`integrity/` (манифест/хеши в
  зеркале), [`configuration.md`](configuration.md)/[`02_recorder_reliability.md`](../promts/02_recorder_reliability.md)
  (deliverable 7 — уточнить контракт зеркала), приёмка `ex_system` (что читается с
  зеркала).

### R-013. Шифрование аудио-сегментов at-rest не подключено в боевой путь записи

- **Статус:** ✅ принято (→ [`promts/13_7_flac_memory_and_segment_encryption.md`](../promts/13_7_flac_memory_and_segment_encryption.md), часть A) · **Дата:** 2026-07-05 · **Автор:** анализ ресурсоёмкости/кода
- **Наблюдение.** Документация (README, `configuration.md` →
  `storage.encrypt_at_rest`, `CLAUDE.md` «ПДн и безопасность») заявляет шифрование
  **записей** at-rest (AES-256-GCM на сегмент). Фактически функция
  `store::crypto::finalize_segment` (SHA-256 по открытому WAV → шифрование в
  `<seg>.enc` → удаление открытой копии) **не имеет ни одного вызова из пути
  записи** — сегменты лежат на диске открытым WAV. Ключом станции шифруются только
  служебные блобы (`auth_session.enc`, `admin_pin.enc`, `operator_profile.enc`,
  `case_cache.enc`). Потребители при этом уже готовы к `.enc`: выгрузка
  (`read_segment_plain` прозрачно дешифрует), плеер/экспорт резолвят ключ;
  заметка о неподключённом live-пути была в `configuration.md` (R-010, «когда
  R-004/этап 13.5 подключит store::crypto в live-путь») — этап 13.5 подключил
  валидацию ключа, но не шифрование сегментов.
- **Ожидаемо.** При `storage.encrypt_at_rest = true` (дефолт) закрытый сегмент
  финализируется `finalize_segment` в потоке райтера (вне аудио-callback'а):
  на диске остаётся только `.enc`; зеркало копирует `.enc`; реконсиляция после
  сбоя дофинализирует «хвостовой» открытый сегмент; отсутствие ключа — громкая
  ошибка записи (fail-secure R-004), а не тихий plaintext. Либо — если решение
  отложится — честно поправить документацию, что at-rest шифруются только
  служебные данные (для системы с ПДн предпочтителен первый вариант).
- **Затрагивает.** `audio/capture.rs` (`run_consumer`, закрытие сегмента),
  `store::crypto` (готово), `store::reconcile` (хеш/дофинализация через
  `read_segment_plain`), `reliability/mirror.rs` (копирует `.enc`),
  `store::retention`, `sync::uploader`/`ipc::player_cmds`/`export` (готовы),
  `configuration.md`/`README`/`admin_guide.md` (сверить формулировки).

### R-014. FLAC-экспорт держит всю сессию в памяти (риск OOM на длинных записях)

- **Статус:** ✅ принято (→ [`promts/13_7_flac_memory_and_segment_encryption.md`](../promts/13_7_flac_memory_and_segment_encryption.md), часть B) · **Дата:** 2026-07-05 · **Автор:** анализ ресурсоёмкости/кода
- **Наблюдение.** `export/flac.rs::encode_wav_to_flac` читает **весь** склеенный
  WAV в `Vec<i32>` (4 байта на семпл) и сериализует FLAC-поток тоже в память
  (`ByteSink`) перед записью на диск. Трёхчасовая моно-сессия 44.1 кГц ≈ 476 млн
  семплов ≈ **1.9 ГБ RAM** (+сжатый выход); многоканал кратно больше. На станции
  с 4 ГБ RAM экспорт длинного заседания в FLAC может уронить приложение по
  памяти. WAV-путь экспорта при этом потоковый (`IntSegmentCursor` — один сегмент
  в памяти) — проблема только у FLAC-ветки.
- **Ожидаемо.** Потоковое/чанковое FLAC-кодирование: пиковая память O(сегмент/блок),
  не O(сессия) — собственный `Source` для `flacenc` поверх посегментного курсора
  (или кодирование фиксированными блоками с дозаписью в файл). Roundtrip-тест
  (`claxon`) сохраняется; `STREAMINFO.total_samples` корректен.
- **Затрагивает.** `src-tauri/src/export/flac.rs` (основное),
  `export/package.rs` (вызов), тесты roundtrip, [`export_package.md`](export_package.md)
  (если меняются оговорки формата), `docs/hardware.md` (снять оговорку о памяти
  после фикса).
