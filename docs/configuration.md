# Конфигурация «Аудиопротокол» — реестр настроек

> **Принцип:** ни одно значение из этого реестра не «зашивается» константой в
> логику. Все параметры — настройки с дефолтами: на станции это типизированная
> схема `Settings` (Rust) + экран «Настройки» + персист в файле конфигурации
> Tauri; на сервере (`ex_system`) — Django settings / env. В коде — только
> обращение к настройке. Этот файл — единый источник истины; промты этапов
> ссылаются на него.

Колонка «Область»: **С** — станция (Tauri-приложение), **Б** — бэкенд
(`ex_system`/Django).

### Многоканальная запись по ролям (фаза 2 — `promts/09_multichannel.md`)

Расширяет захват с одного качественного канала (v1) до **N синхронных дорожек,
привязанных к ролям**. **Аддитивно и выключено по умолчанию:** при
`audio.multichannel.enabled = false` (или пустом `audio.tracks`) поведение —
ровно v1 (один трек `track_id = 0`).

| Параметр | Обл. | Дефолт | Назначение |
|---|---|---|---|
| `audio.multichannel.enabled` | С | `false` | Включение многоканального захвата (иначе — v1: один трек из `audio.device`/`audio.channels`) |
| `audio.tracks` | С | `[]` | Список дорожек `{ device, channel_index, role, label }`; пусто → один трек из `audio.device` |
| `audio.roles` | С | `["judge","clerk","prosecution","defense","witness","room"]` | Справочник ролей дорожек (настраиваемый; роль трека обязана быть из списка) |
| `audio.sync.clock_master_track` | С | `0` | Индекс дорожки — мастера единого клока сессии |
| `audio.sync.drift_threshold_ms` | С | `50` | Порог рассинхрона между дорожками; превышение фиксируется в журнале |
| `audio.sync.drift_compensate` | С | `true` | Компенсация дрейфа (pad/drop семплов) при раздельных устройствах |
| `audio.master_downmix.enabled` | С | `false` | Опц. сведённый мастер (микс всех дорожек) поверх пофайловых дорожек |

`audio.tracks[*].role` ∈ `audio.roles`. `audio.tracks[*].channel_index` — индекс
канала в интерливнутом потоке выбранного `device` (0-based). Один многоканальный
интерфейс (общий клок) → дрейф ≈ 0, компенсация не срабатывает; тот же код
обслуживает и набор USB-микрофонов (раздельный клок, дрейф компенсируется).
Роли дорожек уходят в `ex_system` как вход диаризации W2.11 (контракт — ниже,
«Выгрузка и сеть»).

Подкаталог дорожки на диске (и на зеркале — R-010) именуется
`track-<NN>-<role>`, где `<role>` приводится к **ASCII**: кириллица
транслитерируется (`судья` → `track-00-sudya`) ради читаемости при ручном
разборе носителя и портативности (USB/DVD, FAT/exFAT, ASCII-инструменты). Полное
человекочитаемое имя роли/метки хранится в `tracks.json` **без искажений**; в
логике адресация идёт по `track_id`, поэтому имя каталога на приёмку/выгрузку не
влияет.

## Запись и надёжность

| Параметр | Обл. | Дефолт | Назначение |
|---|---|---|---|
| `recorder.segment_seconds` | С | `30` | Длина сегмента (единица файлов и хеширования) |
| `recorder.flush_interval_ms` | С | `1500` | Интервал fsync — **максимальная потеря при сбое питания** |
| `recorder.max_session_hours` | С | `12` | Защита от «бесконечной» сессии (предупреждение/авто-разбиение) |
| `reliability.watchdog_timeout_ms` | С | `5000` | Таймаут watchdog до авто-рестарта записи |
| `reliability.disk_low_threshold_mb` | С | `1024` | Порог предупреждения о свободном месте |
| `reliability.disk_critical_mb` | С | `256` | Критический порог (защитные действия) |
| `reliability.device_reconnect.auto_resume` | С | `true` | Авто-возобновление при возврате устройства |
| `reliability.mirror.enabled` | С | `false` | Дублирующая дорожка на второй носитель |
| `reliability.mirror.path` | С | — | Корневой путь второго носителя; зеркало **повторяет структуру** основного хранилища (см. ниже) |

**Структура зеркала (R-010, этап 13.3).** Зеркало — не плоская свалка аудио, а
**копия дерева основного места**: `<mirror.path>/<session>/<track>/seg-*.wav`
(так же, как `storage.root_path`). Индекс сегмента идёт с 1 **на дорожку**, поэтому
в едином плоском каталоге имена дорожек/сессий совпадали бы и `std::fs::copy` тихо
перезаписывал файл (потеря данных) — структура это исключает. Рядом с аудио
зеркалятся **метаданные** (журнал(ы) `session.journal` + `tracks.json`),
минимально достаточные, чтобы по зеркалу **независимо** реконсилировать сессию
([`store::reconcile`]) — SQLite-манифест и целостность (SHA-256/хеш-цепочка)
пересчитываются из журнала + сегментов, дублировать их на зеркало не нужно — и
выгрузить запись в `ex_system`. Сегменты копируются «на лету» по факту закрытия
(при `storage.encrypt_at_rest` — **после** финализации, этап 13.7: на зеркало
едет тот же `.enc`-блоб, plaintext не живёт на втором носителе дольше основного);
метаданные — при старте (реконсилируемость и после сбоя носителя) и при штатном
стопе. Файлы копируются **как есть** (байт-в-байт): шифрование at-rest
наследуется от основного места — своего криптопути у зеркала нет, plaintext
поверх шифрованного оно не пишет. Зеркалирование остаётся
**best-effort**: сбой (нет места/носитель отвалился) только логируется и **не
влияет** на основную запись. `reliability.mirror.path` — машинно-зависимый путь
(см. таблицу ниже): при тиражировании профиля станции очищается.

## Хранилище и ретеншн (локально на станции)

| Параметр | Обл. | Дефолт | Назначение |
|---|---|---|---|
| `storage.root_path` | С | `<data-dir>/recordings` | Корень локального хранилища |
| `storage.encrypt_at_rest` | С | `true` | Шифрование записей на диске (AES-256-GCM на сегмент) |
| `storage.key_source` | С | `os_keystore` | Источник ключа шифрования at-rest |
| `retention.mode` | С | `until_confirmed_plus_window` | Политика удаления локальной копии |
| `retention.require_integrity_verified` | С | `true` | Удалять только после серверного `integrity_verified=true` |
| `retention.safety_window_hours` | С | `72` | Буфер после подтверждения до удаления |

`retention.mode` ∈ { `until_confirmed_plus_window` (дефолт), `delete_on_confirm`
(минимизация ПДн), `manual` (до ручного удаления) }.

`storage.key_source` ∈ { `os_keystore` (дефолт), `passphrase` }. Секрет в файле
настроек **не хранится**. Для `passphrase` (а также как фолбэк, пока OS-keystore
не интегрирован — этап `08`) ключ — Argon2id из парольной фразы станции,
переданной через env `COURT_AUDIO_STATION_PASSPHRASE`; соль персистится в
`<root_path>/key.salt`. Параметры Argon2id — рекомендованные дефолты крейта
(крипто-константы, не бизнес-логика).

**Ключ станции обязателен и провижинится при развёртывании (R-004, этап 13.5).**
Тем же ключом шифруются кэш офлайн-сессии (`auth_session.enc`), админ-PIN
(`admin_pin.enc`), кэш дел (`case_cache.enc`) и — при `storage.encrypt_at_rest` —
аудио-сегменты. Без ключа эти блобы **не пишутся** — раньше сбой проглатывался
молча (онлайн-вход «успешен», а офлайн-старт по PIN в следующий раз недоступен).
Теперь наличие ключа **валидируется на старте** (лог + пункт self-test «Ключ
станции»), сбой шифрования доводится до UI явной ошибкой, а не теряется. Порядок
провижининга (рядом с `COURT_AUDIO_ADMIN_PIN`) — [`packaging.md`](packaging.md).

### Машинно-зависимые настройки (перенос конфига, R-005)

Часть параметров привязана к **конкретной машине/ОС** и не переносится дословно
между станциями:

| Параметр | Почему не переносится |
|---|---|
| `audio.device` | Имя устройства ввода уникально для ОС/машины (напр. `«Микрофон MacBook Air»` не существует на Windows) |
| `storage.root_path` | Абсолютный путь к локальному хранилищу (разные диски/тома) |
| `reliability.mirror.path` | Абсолютный путь к дублирующему носителю |

При тиражировании профиля станции на другой зал **очищайте** эти поля (значение
по умолчанию: `audio.device = null` → системное устройство; пустые пути → системный
data-каталог), иначе:

- отсутствующее `audio.device` даёт «молчаливую тишину» индикатора/захвата.
  Экран «Запись» и self-test теперь **явно предупреждают** («выбранный микрофон не
  найден — проверьте настройки») и мягко откатываются на системное устройство —
  но правильнее чистить поле при переносе;
- несуществующий `storage.root_path`/`mirror.path` ломает запись/зеркало.

Экспорт/импорт профиля станции (этап 10.4) — путь тиражирования; чистку
машинно-зависимых полей выполняет администратор при импорте. См. также
[`packaging.md`](packaging.md).

## Выгрузка и сеть

| Параметр | Обл. | Дефолт | Назначение |
|---|---|---|---|
| `sync.server_base_url` | С | — (обяз.) | Базовый URL `ex_system` |
| `sync.chunk_size_mb` | С | `8` | Размер чанка возобновляемой загрузки |
| `sync.parallel_uploads` | С | `1` | Параллельных выгрузок |
| `sync.auto_upload` | С | `true` | Авто-выгрузка по готовности записи (фон) |
| `sync.defer_during_recording` | С | `false` | Откладывать догрузку до конца активной сессии |
| `sync.offline_queue.enabled` | С | `true` | Оффлайн-очередь выгрузки |
| `sync.retry.backoff_base_ms` | С | `2000` | База экспоненциального бэкоффа |
| `sync.retry.backoff_max_ms` | С | `60000` | Потолок бэкоффа |
| `sync.retry.max_attempts` | С | `0` (без лимита) | Лимит попыток (0 = бесконечно, до успеха) |
| `AUDIO_MAX_UPLOAD_PART_MB` | Б | `16` | Серверный лимит размера части |

**Размер части ↔ размер сегмента (важно).** Часть возобновляемой загрузки = **целый
сегмент** (клиент сегмент не дробит: `part/<track_id>/<part_index>` шлёт один
файл сегмента). Отсюда обязательное соотношение:

```
размер_сегмента ≤ sync.chunk_size_mb ≤ AUDIO_MAX_UPLOAD_PART_MB
```

Размер сегмента (WAV-PCM) ≈ `recorder.segment_seconds × audio.sample_rate_hz ×
audio.channels × audio.bit_depth / 8` байт (напр. 30 с × 44.1 кГц × 1 × 16 бит ≈
2.6 МБ — с дефолтами укладывается в оба лимита). При увеличении
`recorder.segment_seconds` (или `sample_rate_hz`/`channels`/`bit_depth`) сегмент
растёт **кратно**, и оба лимита нужно поднять согласованно, иначе:

- сегмент > `sync.chunk_size_mb` → станция валит выгрузку **до сети**
  (`SyncError::Permanent` «часть N больше предела sync.chunk_size_mb», статус
  `Failed`, в логах backend — ни одного запроса части);
- сегмент > `AUDIO_MAX_UPLOAD_PART_MB` → сервер отвечает `413` на `part`.

Пример (300-секундный сегмент моно 44.1 кГц/16 бит ≈ 25.2 МиБ): поставить
`sync.chunk_size_mb = 32` и `AUDIO_MAX_UPLOAD_PART_MB = 32`. `AUDIO_MAX_UPLOAD_PART_MB`
на сервере задаётся через env (`config(..., default=16)`; в dev — `compose.yml`
сервиса `backend`). Обратная сторона больших частей: сервер читает часть целиком
в память (~размер сегмента на загрузку); **на проде** поднимите
`client_max_body_size` в nginx до `≥ AUDIO_MAX_UPLOAD_PART_MB`. Короткие сегменты
предпочтительнее и по «бесперебойности» (потеря максимум незавершённого сегмента).

**Мультитрек-контракт (фаза 2, этап 09).** Одна `AudioRecording` на сессию + N
треков внутри (`tracks[] = { track_id, role, label, final_chain_link,
segments[] }`). Часть адресуется по треку:
`PUT /audio/recordings/{id}/upload/part/{track_id}/{part_index}/`. Роли доходят
до диаризации W2.11. Серверный приём — этап `07` в `ex_system` (правится
синхронно с `promts/06_sync_agent.md`).

**Контракт разметки (фаза 2, этап 10, `manifest_version = 3`).** Манифест несёт
блок `annotations = { chain_final_link, markers[], role_spans[], log[] }`:
`log[]` — append-only лог действий под хеш-цепочкой (для серверной проверки
целостности разметки), `markers[]`/`role_spans[]` — свёрнутое текущее состояние
(подсказки W2.11). Разметка передаётся в `POST /audio/recordings/{id}/upload/init/`
рядом с `tracks`: `{ "tracks": [...], "annotations": {...} }`. Приём и
проброс в W2.11 — этап `07` в `ex_system` (правится синхронно).

## Разметка: метки и роли (фаза 2 — `promts/10_markers_realtime.md`)

Ручная **живая разметка** заседания: закладки с таймкодом (категория + опц.
комментарий) и интервалы «сейчас говорит <роль>». Разметка — подсказки для
диаризации/протокола W2.11 в `ex_system`; ASR/NER здесь не дублируется.
Действия идут **вне аудио-потока** (не мешают захвату), пишутся в write-ahead
журнал сессии (переживают рестарт), под хеш-цепочкой (tamper-evident,
авторство). **Аддитивно:** пустой справочник — просто нет быстрых кнопок.

| Параметр | Обл. | Дефолт | Назначение |
|---|---|---|---|
| `markers.categories` | С | `["Закладка","Инцидент","Перерыв","Прочее"]` | Справочник категорий закладок (человекочитаемые ярлыки; значение уходит в W2.11 как подсказка). Настраивается заказчиком — согласован минимальный набор |

Роли интервалов **переиспользуют справочник `audio.roles`** (нового параметра
нет). В многоканале (этап 09) роль интервала может подставляться из активной
дорожки; ручная корректировка — выбор любой роли из `audio.roles`. Смещения
метки/интервала семпл-точны относительно **общей оси сессии** (единый
`started_at` + частота записи).

## Проигрыватель (улучшение после v1 — `promts/10_1_playback.md`)

Встроенный проигрыватель завершённых сессий: потоковая дешифровка сегментов в
памяти ядра (ключ не покидает Rust), навигация по меткам/интервалам ролей
(`10`), выбор дорожки/микс (`09`). Вывод звука — из Rust (`rodio` поверх уже
используемого `cpal`), plaintext не пересекает границу Tauri IPC.

| Параметр | Обл. | Дефолт | Назначение |
|---|---|---|---|
| `player.seek_step_seconds` | С | `15.0` | Шаг перемотки кнопками ±N с (заседание идёт часами — компромисс между точностью попадания в реплику и числом нажатий) |
| `player.playback_rates` | С | `[0.5, 0.75, 1.0, 1.25, 1.5, 2.0]` | Доступные скорости воспроизведения (диапазон 0.5–2×, стандартный набор аудио/видео-плееров) |
| `player.position_update_hz` | С | `5` | Частота события позиции `player_position` для UI; ниже `audio.level_update_hz` — плейхеду на многочасовой шкале не нужна живость метра реального времени |

## Экспорт (улучшение после v1 — `promts/10_2_export.md`)

Выдача самодостаточной копии записи (участнику процесса/в архив/на носитель):
дешифрованное аудио (WAV/FLAC) + метки/роли + манифест целостности копии +
автономный HTML-плеер, в папку (в т.ч. USB) или на data-DVD. Опирается на
движок дешифровки/склейки этапа `10.1` (побайтовый, не потоковый путь — см.
`docs/export_package.md`). Экспорт выводит ПДн из зашифрованного хранилища —
управляемое настройкой администратора и всегда журналируемое действие
(`EventKind::ExportCreated`, в т.ч. отказ политикой).

| Параметр | Обл. | Дефолт | Назначение |
|---|---|---|---|
| `export.policy` | С | `allowed` | Политика администратора: `allowed` / `forbidden` / `requires_confirmation` |
| `export.default_codec` | С | `wav_pcm` | Предустановленный формат в мастере экспорта (`wav_pcm`/`flac`); оператор может выбрать другой формат прямо в мастере |

Промт `10_2_export.md` называл кандидатами два независимых параметра
(`export.enabled`, `export.require_confirmation`); в реализации они объединены
в один `export.policy` — исключает противоречивую комбинацию (`enabled=false`
при одновременном `require_confirmation=true`) и ложится 1:1 на трёхвариантный
критерий приёмки промта («экспорт разрешён / запрещён / только с
подтверждением»).

## Администрирование: разграничение доступа (этап 10.4 — `promts/10_4_settings_roles.md`)

Разделяет настройки на **оператор-скоуп** (рабочие параметры) и **админ-скоуп**
(инфраструктура/безопасность). Оператор админ-секцию видит, но правит только
свою; попытка изменить админ-параметр отклоняется **на уровне ядра** (IPC), а
не только в UI. Права администратора в v1 — **валидный админ-PIN** (оффлайн-
фолбэк; работать в оффлайне обязательно). Роль-из-`ex_system` как источник прав
(`admin.access`) — отложена (открытый вопрос промта, согласуется с `ex_system`).

| Параметр | Обл. | Дефолт | Назначение |
|---|---|---|---|
| `admin.pin.required` | С | `true` | Требовать админ-PIN для изменения админ-настроек (гейт ядра; `false` — все настройки правит оператор) |
| `admin.pin.min_length` | С | `4` | Минимальная длина админ-PIN (валидация при провижининге) |

Сам PIN в реестре/`settings.json` **не хранится**: задаётся при развёртывании
через env `COURT_AUDIO_ADMIN_PIN`, на первом запуске персистится как Argon2id-
хеш в зашифрованном блобе `admin_pin.enc` (ключ станции, как кэш оффлайн-сессии
`10.3`); дальше проверка идёт оффлайн против блоба, env можно снять. Если PIN не
задан при развёртывании — админ-изменения отклоняются (оператор-настройки
правятся всегда). Каждое сохранение настроек — событие журнала (кто/когда/что:
старое→новое, без секретов) при `integrity.event_log`; опасные изменения (смена
`sync.server_base_url`, отключение `storage.encrypt_at_rest`, смягчение
`retention.mode`/`retention.require_integrity_verified`) — после явного
подтверждения. Профиль станции (полный `Settings` в JSON, секретов там нет)
экспортируется/импортируется для тиражирования на залы; импорт — только
администратором, с журналом.

### Область доступа (оператор/админ)

**Оператор-скоуп** (правит на экране «Настройки»): `audio.device`, `audio.roles`,
`markers.categories`, `player.*` (устройство, справочники меток/ролей, удобства).

**Админ-скоуп** (правит на экране «Администрирование» после разблокировки PIN):
**всё остальное** — сервер/URL (`sync.*`), шифрование/ретеншн (`storage.*`,
`retention.*`), пороги надёжности (`recorder.*`, `reliability.*`), карта дорожек
многоканала (`audio.multichannel`/`audio.tracks`/`audio.sync`/
`audio.master_downmix` и параметры качества `audio.sample_rate_hz`/`bit_depth`/
`channels`/`master_codec`/`capture_buffer_seconds`/`level_update_hz`/
`archive_copy`), целостность (`integrity.*`), политика экспорта (`export.*`),
политика аутентификации (`auth.*`), кэш дел (`case_cache.*`), интерфейс станции
(`ui.*` — доступность режима зала/оверлея), звуковой сигнал о сбоях (`ux.*` —
этикет зала суть политика станции), сам блок `admin.*`.

## Интерфейс и адаптивность (этап 10.5 — `promts/10_5_ui_adaptive.md`)

Адаптивная сетка (сворачивание боковой навигации на узких окнах), глобальный
индикатор записи (шапка + системный трей), «режим зала» с крупным статусом и
опц. компакт-окно «поверх всех окон» (Tauri multi-window). Брейкпоинты сетки и
косметика метров — **layout-константы кода** (не бизнес-параметры реестра, как
`METER_FLOOR_DBFS` и т.п. в экране «Запись»); реестр несёт только доступность
режима зала и оверлея.

| Параметр | Обл. | Дефолт | Назначение |
|---|---|---|---|
| `ui.hall_mode.enabled` | С | `true` | Доступность «режима зала» (крупный статус, читаемый с нескольких метров) |
| `ui.compact_overlay.enabled` | С | `false` | Доступность компакт-окна статуса «поверх всех окон» (оператор работает в других программах, не теряя контроль записи) |

## UX-пакет (этап 10.6 — `promts/10_6_ux_polish.md`)

Проверка перед заседанием (self-test), мастер первого запуска, поиск/карточка
сессии, защита идущей записи от «сюрпризов» ОС, уведомления о сбоях, мелочи
трения. Self-test и мастер **переиспользуют** существующие пороги (`reliability.*`,
`sync.*`, `auth.*`) — новых параметров не вводят. Единственный параметр реестра —
опциональный звуковой сигнал.

| Параметр | Обл. | Дефолт | Назначение |
|---|---|---|---|
| `ux.sound_alerts.enabled` | С | `false` | Опциональный звуковой сигнал при сбоях (обрыв устройства / критический диск / ошибка выгрузки). **Дублирует, не заменяет** баннеры/трей. Выключен по умолчанию: в зале звук может быть неуместен (этикет; согласовано — тишина по умолчанию, оператор включает при необходимости) |

### ОС-интеграция (вне реестра приложения)

**Блокировка сна/гибернации** во время записи и **автозапуск приложения при
входе в систему** реализуются **на уровне ОС** при развёртывании (этап `08`), а не
параметрами/кодом приложения (чтобы не плодить неработающие тумблеры-заглушки).
Платформенные рекомендации (Astra Linux SE / РЕД ОС / Windows) — на экране
«Администрирование» и в [`os_integration.md`](os_integration.md). App-side без
плагинов реализованы: подтверждение при **закрытии окна с идущей записью**
(core-API `onCloseRequested`) и опциональный звуковой сигнал (`ux.sound_alerts`,
Web Audio в webview). Фоновые системные нотификации при свёрнутом окне требуют
плагина ОС-уведомлений и остаются перспективой (см. `os_integration.md`).

## Аутентификация

Решение заказчика: **оператор обязан авторизоваться**. Нюанс надёжности —
вход требуется для **старта** сессии; идущую запись истечение токена/выход не
прерывает; в оффлайн-зале старт идёт по **кэшированной** сессии оператора.

| Параметр | Обл. | Дефолт | Назначение |
|---|---|---|---|
| `auth.station_identity.required` | С/Б | `true` | У станции своя учётка на зал (транспорт выгрузки) |
| `auth.operator.required_to_start` | С/Б | `true` | Оператор обязан войти перед стартом записи |
| `auth.operator.cached_session_hours` | С | `24` | Срок кэш-сессии для оффлайн-старта |
| `auth.operator.offline_pin.required` | С | `true` | Требовать PIN для оффлайн-старта по кэшированной сессии (второй фактор на станции) |
| `auth.operator.offline_pin.min_length` | С | `4` | Минимальная длина PIN (валидация при установке на входе) |
| `auth.operator.autonomous_offline.enabled` | С | `false` | Автономный офлайн-старт по провижиненному PIN в **изолированном зале** (B-001, этап 13.6). Ослабляет требование онлайн-входа, поэтому **только для явно провижиненных залов** и выключено по умолчанию — обычные станции вход не ослабляют. В изолированном зале через UI не включить (экран «Администрирование» за гейтом входа) — провижинится env `COURT_AUDIO_AUTONOMOUS_OFFLINE=1` на первом старте или готовым `settings.json` (см. [`first_run_offline.md`](first_run_offline.md)) |
| `auth.recording_survives_token_expiry` | С | `true` | Идущая запись не прерывается при истечении токена |

В модели `AudioRecording` (Б) фиксируются `station_id` **и** обязательный
`operator_id`.

**Клиент аутентификации (этап 10.3).** Вход — JWT-flow `ex_system`
(`docs/auth.md` в `ex_system`): `POST /api/token/` (`{email,password}` →
`{access,refresh}`), `POST /api/token/refresh/` (`{refresh}` → `{access}`),
профиль оператора — `GET /user/` (`id` → числовой `operator_id`, ФИО, роль).
**Кэш оффлайн-сессии** (`store/`, зашифрован AES-256-GCM ключом станции) хранит
`operator_id`/ФИО/роль + **refresh-токен** + хеш PIN (Argon2id) в пределах
`cached_session_hours`; возврат онлайн — тихий refresh без действий оператора.
Секреты (пароль, access/refresh, PIN) в `settings.json` **не хранятся**. Детали —
[`auth.md`](auth.md).

**Автономный офлайн-старт (B-001, этап 13.6).** При
`auth.operator.autonomous_offline.enabled = true` в **изолированном зале** (нет
`ex_system` и не было ни одного онлайн-входа) станция стартует по
**провижиненному** операторскому профилю зала: `operator_id`/`station_id`/ФИО/роль
+ хеш PIN (Argon2id) — в зашифрованном блобе `operator_profile.enc` (ключ станции,
как `admin_pin.enc`/кэш-сессия). Идентичность берётся из профиля, а не из
онлайн-сессии; разблокировка **оффлайн** по PIN (контур `store::operator_profile`,
образец — админ-PIN). Провижининг — env рядом с `COURT_AUDIO_ADMIN_PIN`
([`packaging.md`](packaging.md)); сам PIN — только хешем, в `settings.json` не
хранится. Флаг **выключен по умолчанию**: обычные станции профиль не создают и
вход не ослабляют. Запись помечается признаком `autonomous_offline` (контракт
`07`); выгрузка — через существующую отложенную очередь. Модель идентичности и
принятые решения по открытым вопросам — [`auth.md`](auth.md).
