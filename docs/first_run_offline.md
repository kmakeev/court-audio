# Первый запуск в полностью оффлайн-режиме

Инструкция администратора по вводу станции «Аудиопротокол» в эксплуатацию
**без доступа к `ex_system`** — от установки дистрибутива до первой записи.
Модель аутентификации — [`auth.md`](auth.md); провижининг секретов —
[`packaging.md`](packaging.md); реестр настроек — [`configuration.md`](configuration.md).

## Два оффлайн-сценария

Станция различает «временно нет связи» и «связи не будет вообще»:

| Сценарий | Первый вход | Дальнейшие старты |
|---|---|---|
| **Обычный зал, связь бывает** | Хотя бы один **онлайн-вход** оператора (JWT `ex_system`) — он кэширует сессию и хеш PIN | Оффлайн-старт по кэшированной сессии + PIN в окне `auth.operator.cached_session_hours` (дефолт 24 ч) |
| **Изолированный зал** (`ex_system` недоступен никогда) | **Автономный старт по провижиненному PIN** (B-001, этап 13.6) — онлайн-вход не требуется ни разу | Тот же автономный старт по PIN; идентичность — из провижининг-профиля зала |

Автономный режим ослабляет правило «оператор обязан авторизоваться онлайн»,
поэтому включается **только** флагом `auth.operator.autonomous_offline.enabled`
(дефолт «выкл.») для явно провижиненных залов — обычные станции вход не
ослабляют. Решение согласовано с заказчиком (см. [`auth.md`](auth.md), B-001).

## Предпосылки

1. **Дистрибутив установлен оффлайн.** Пакеты `.deb`/`.rpm` (Astra SE / РЕД ОС)
   или NSIS-инсталлятор Windows с **фикс-версией WebView2** в бандле — интернет
   при установке не нужен. Форматы, зависимости и подпись пакетов —
   [`packaging.md`](packaging.md).
2. **ОС подготовлена.** Отключён сон/гибернация на время записи, настроен
   автозапуск приложения при входе в систему — средствами ОС по инструкции
   [`os_integration.md`](os_integration.md).

## Провижининг секретов (до первого старта)

Секреты **не хранятся** в `settings.json` — задаются переменными окружения в
скрипте запуска/systemd-юните станции **до** старта приложения (детали и
таблицы — [`packaging.md`](packaging.md), раздел «Провижининг секретов станции»):

| Env | Назначение | Когда нужна |
|---|---|---|
| `COURT_AUDIO_STATION_PASSPHRASE` | Парольная фраза → ключ станции (Argon2id): шифрует аудио-сегменты (при `storage.encrypt_at_rest`, этап 13.7) и служебные блобы (кэш сессии, PIN-хеши, кэш дел) | **Всегда, при каждом старте** (R-004) |
| `COURT_AUDIO_ADMIN_PIN` | Админ-PIN → хеш в `admin_pin.enc` | Первый запуск; после засева можно снять |
| `COURT_AUDIO_STATION_ID` | Идентификатор станции (учётка транспорта выгрузки) | Всегда |
| `COURT_AUDIO_OPERATOR_PIN` | PIN оператора зала → хеш в `operator_profile.enc` | Изолированный зал; после засева можно снять |
| `COURT_AUDIO_OPERATOR_ID` | `operator_id` провижиненного оператора зала — **числовой PK реального пользователя `ex_system`** (сервер резолвит учётку; неизвестный id → `400`, выгрузка зала встанет) | Изолированный зал, при засеве |
| `COURT_AUDIO_OPERATOR_NAME` / `_ROLE` | ФИО и роль оператора зала (шапка/журнал) | Опционально, при засеве |
| `COURT_AUDIO_AUTONOMOUS_OFFLINE` | **Не секрет.** `1`/`true` — на первом старте включает автономный режим (пишет `settings.json` с `auth.operator.autonomous_offline.enabled = true`), см. «Порядок» ниже | Изолированный зал, первый старт |

**Фраза станции должна быть стабильной между рестартами**: соль KDF персистится
в `<storage.root_path>/key.salt`, при смене фразы ключ не воспроизведётся и
зашифрованные блобы (кэш сессии, PIN-хеши, кэш дел, сегменты) перестанут
читаться.

## Порядок первого запуска изолированного зала

Профиль зала засеивается ядром **на старте** и **только при включённом флаге**
автономного режима. Но включить флаг через UI в изолированном зале **нельзя**:
экран «Администрирование» — за гейтом входа оператора, а войти без флага (и без
онлайн-связи) не выйдет — «курица и яйцо». Поэтому флаг задаётся до/на первом
старте, **вне UI**, одним из двух способов ниже. Оба включают режим уже на
**первом** старте — перезапуск не нужен.

### Способ A (рекомендуемый) — через env

Флаг провижинится тем же путём, что ключ станции/админ-PIN/профиль оператора:

1. Выставить env из таблицы выше, **включая** `COURT_AUDIO_AUTONOMOUS_OFFLINE=1`
   (скрипт запуска / systemd-юнит).
2. **Первый старт приложения.** Ядро видит, что `settings.json` ещё нет и флаг в
   env взведён → пишет дефолтные настройки с
   `auth.operator.autonomous_offline.enabled = true`, валидирует ключ станции,
   засеивает админ-PIN (`admin_pin.enc`) и — раз флаг уже включён — операторский
   профиль зала в `operator_profile.enc` (`operator_id`/`station_id`/ФИО/роль +
   Argon2id-хеш PIN). В лог пишется строка `автономный офлайн-режим включён из
   env`. Профиль **не** засеется, если PIN короче
   `auth.operator.offline_pin.min_length` (дефолт 4) или не заданы
   `COURT_AUDIO_OPERATOR_PIN`/`_ID`/`STATION_ID` — ошибка в логе, тихого
   пропуска нет.
3. На экране входа появляется карточка **«Автономный старт»**: оператор вводит
   PIN — активируется оффлайн-сессия с идентичностью из профиля, доступен старт
   записи.
4. После успешного засева `COURT_AUDIO_ADMIN_PIN`, `COURT_AUDIO_OPERATOR_PIN` и
   `COURT_AUDIO_AUTONOMOUS_OFFLINE` из окружения можно снять (флаг уже в
   `settings.json`, проверка PIN дальше идёт оффлайн против блобов);
   `COURT_AUDIO_STATION_PASSPHRASE` и `COURT_AUDIO_STATION_ID` остаются.

> **Идемпотентность.** Env-провижининг флага срабатывает **только при
> отсутствующем** `settings.json` — на уже настроенной станции файл не
> перезаписывается, случайно «включить» режим переменной окружения нельзя.

### Способ B — готовый `settings.json` (тиражирование)

Если удобнее разложить конфиг файлом (например, образом на несколько залов) —
положите **до первого старта** валидный `settings.json` с включённым флагом в
каталог конфигурации приложения:

| ОС | Путь к `settings.json` |
|---|---|
| Windows | `%APPDATA%\ru.court.audioprotocol\settings.json` |
| Astra SE / РЕД ОС | `~/.config/ru.court.audioprotocol/settings.json` (или `$XDG_CONFIG_HOME/ru.court.audioprotocol/`) |

Готовый образец с уже включённым `auth.operator.autonomous_offline.enabled` —
[`samples/settings.autonomous_offline.json`](samples/settings.autonomous_offline.json)
(дефолты реестра v0.2.0 + один включённый флаг). Скопируйте его под именем
`settings.json`, при необходимости отредактируйте другие ключи по
[`configuration.md`](configuration.md), затем стартуйте — дальше шаги 2–4 способа
A (env оператора/станции всё равно нужны для засева профиля и ключа станции).
Альтернатива для парка залов — импорт **профиля станции** с включённым флагом
(экспорт/импорт — этап 10.4; машинно-зависимые поля при импорте очищаются).

## Проверка перед первым заседанием

Запустить **self-test** (экран «Диагностика» / проверка перед заседанием) и
убедиться:

- [ ] **«Ключ станции»** — OK. `Fail` означает, что фраза не задана/не та:
      запись при `storage.encrypt_at_rest` (дефолт) не сохранится.
- [ ] Микрофон найден, уровень на метрах живой (сказать несколько слов).
- [ ] Свободное место на диске выше порогов `reliability.disk_*`.
- [ ] Пробная запись: старт по PIN → 1–2 минуты → стоп → сессия видна в списке,
      сегменты и журнал на месте.
- [ ] После перезапуска приложения автономный старт по PIN работает **без** env
      с PIN (проверка идёт против блоба).

## Работа в изолированном зале

- **Привязка к делу** — только **ручной ввод №** дела (механизм этапа 05):
  докет из `ex_system` недоступен, предзагрузка кэша дел для автономного зала
  не вводится (решение заказчика, [`auth.md`](auth.md)).
- **Выгрузка** копится в отложенной оффлайн-очереди (`sync.offline_queue`).
  Записи автономной сессии помечаются признаком `autonomous_offline` — при
  появлении связи сервер принимает их по ключу станции-транспорта и помечает
  для аудита/последующей сверки (контракт этапа 07).
- **Ретеншн**: локальная копия по умолчанию удаляется только после
  подтверждённой выгрузки (`retention.mode = until_confirmed_plus_window`) — в
  изолированном зале записи хранятся на станции, пока их не заберут
  (выгрузкой при появлении связи или экспортом на носитель, этап 10.2).
- **Идущую запись** истечение сессии/смена оператора не прерывает — гейт
  авторизации действует только на старт новой сессии.

## Частые проблемы

| Симптом | Причина и действие |
|---|---|
| Self-test «Ключ станции» = Fail; в логе «ключ станции недоступен» | Не выставлена/изменена `COURT_AUDIO_STATION_PASSPHRASE`. Вернуть исходную фразу; при утрате фразы старые блобы и зашифрованные сегменты не восстановить |
| Карточки «Автономный старт» нет на экране входа | Флаг `autonomous_offline.enabled` не включён (не задан `COURT_AUDIO_AUTONOMOUS_OFFLINE=1` **до первого старта**, либо `settings.json` уже существовал и env-провижининг его не тронул), либо профиль не засеян (см. лог первого старта: PIN короче минимума или неполные env). На уже стартовавшей чистой станции env флаг не включит — используйте способ B (готовый `settings.json`) или удалите `settings.json` и стартуйте заново с env |
| «Конфигурация повреждена — обратитесь к администратору» | Битый `settings.json`: гейты смыкаются fail-secure (R-003) — старт записи и изменение настроек заблокированы. Восстановить/удалить файл настроек (при отсутствии файла применяются безопасные дефолты) |
| Онлайн-вход «успешен», а оффлайн-старт потом недоступен | На станциях со связью это признак отсутствия ключа станции — с этапа 13.5 такой вход завершается явной ошибкой, а не тихим успехом; проверить фразу |

## Безопасность автономного режима

- PIN хранится **только хешем** (Argon2id, случайная соль, сравнение
  константное по времени); в `settings.json` секретов нет.
- Блоб `operator_profile.enc` зашифрован ключом станции (AES-256-GCM) с
  fail-secure реакцией на отсутствие ключа — сбой не проглатывается.
- Флаг режима — админ-скоуп: оператор изменить его не может (гейт на уровне
  ядра, этап 10.4).
- Все старты/стопы, идентичность (`operator_id`, `station_id`) и изменения
  настроек фиксируются в журнале событий под хеш-цепочкой.
