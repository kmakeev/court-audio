# Рекомендации по развитию «Аудиопротокола»

Снимок предложений на **2026-07-02**. Фокус ближайшего цикла — **функциональность
и UI/UX**; этапы `11_gost_signing` и `12_video` остаются перспективой (промты
готовы, реализация не начата), серверный приём (этап `07` в `ex_system`)
реализован частично и дозавершается параллельно.

Исходное состояние: v1 (этапы 00–08) завершён, реализованы также `09`
(многоканал) и `10` (живая разметка). Ниже — направления улучшений с текущим
состоянием и ожидаемым результатом. Нумерация — в порядке рекомендуемого
приоритета.

---

## 1. Встроенный проигрыватель записей с навигацией по меткам

**Что сейчас.** Воспроизведения нет вообще: записанную сессию нельзя прослушать
ни на станции, ни при выдаче. Записи зашифрованы at-rest (AES-256-GCM по
сегментам), поэтому «открыть WAV во внешнем плеере» невозможно в принципе.
Живая разметка (этап 10) создаёт метки и интервалы ролей, но после остановки
записи их негде увидеть — ценность разметки теряется до попадания в `ex_system`.

**Что сделать.**
- Команды ядра для воспроизведения: потоковое чтение сегментов с дешифровкой
  «на лету» (ключ не покидает ядро), склейка по манифесту, отдача в аудио-выход
  (`cpal`/`rodio`) либо стрим в webview через кастомный протокол Tauri.
- Экран/панель «Прослушивание», открываемая из карточки сессии:
  - таймлайн всей записи с нанесёнными **метками** (категория + комментарий)
    и **интервалами ролей**; клик по метке — мгновенный переход;
  - список меток рядом с таймлайном (как оглавление записи);
  - управление: play/pause, перемотка ±10 с, скорость 0.5–2×, громкость,
    горячие клавиши;
  - для многоканальных сессий — выбор дорожки (по роли) или сведённый микс;
  - индикация «прослушивается фрагмент N сегмента M» и статуса целостности.
- Прослушивание — событие журнала (`integrity.event_log`): кто и когда слушал
  (аудит доступа к ПДн).
- Прослушивание не должно мешать идущей записи (отдельный поток, приоритет
  захвата выше).

**Ожидаемый результат.** Секретарь сразу после заседания проверяет запись и
находит нужные фрагменты по меткам за секунды; разметка этапа 10 становится
полезной локально, а не только как подсказка W2.11. Проигрыватель — общий
фундамент для экспорта (п. 2).

## 2. Экспорт записей: в папку, на DVD, автономное прослушивание

**Что сейчас.** Наружу выдаётся только JSON-манифест (серверная верификация).
Аудио с станции можно получить единственным путём — выгрузкой в `ex_system`.
Штатного сценария «выдать копию записи участнику процесса» или «сдать в архив
на носителе» нет, хотя для судов это типовое требование (ориентир — SRS
«Фемида»).

**Что сделать.**
- **Мастер экспорта** из карточки сессии: выбор состава (все дорожки / микс /
  одна дорожка), формата (WAV без потерь / FLAC для экономии места) и
  назначения (папка, USB-носитель, DVD).
- Состав экспортного пакета:
  - аудиофайлы (дешифрованные, склеенные из сегментов);
  - метки и интервалы ролей (JSON + человекочитаемый список; опц. CUE);
  - манифест целостности: SHA-256 файлов, параметры записи, дело, дата,
    станция/оператор;
  - **автономный HTML-плеер** — одна страница без зависимостей, открывается
    с диска в любом браузере: таймлайн, переход по меткам, регулятор скорости.
    Получатель копии слушает запись без установки какого-либо ПО.
- **Запись на DVD** (data-DVD): через системные средства — IMAPI (Windows),
  `growisofs`/аналог (Astra/РЕД ОС); после прожига — верификация чтением и
  сверкой хешей. DVD содержит тот же пакет, включая HTML-плеер.
- Аудит и контроль: каждый экспорт — событие журнала (кто, когда, какая сессия,
  куда); настройка администратора «экспорт разрешён/запрещён/только с
  подтверждением» (ПДн покидают зашифрованное хранилище — это управляемое
  действие).
- Все параметры (формат по умолчанию, разрешение экспорта, путь) — через реестр
  `docs/configuration.md`, без «магических чисел».

**Ожидаемый результат.** Станция закрывает полный жизненный цикл записи без
`ex_system`: выдача заверенной копии стороне процесса, сдача в архив на
носителе, прослушивание копии на любом компьютере. Целостность копии проверяема
(хеши в манифесте пакета).

## 3. Интеграция авторизации с `ex_system` (онлайн-работа)

**Что сейчас.** Экрана входа нет; операторский токен подаётся через env
`COURT_AUDIO_OPERATOR_TOKEN` (в коде помечено «до экрана входа»). Политика уже
зафиксирована в `docs/configuration.md` (`auth.operator.required_to_start`,
кэш-сессия 24 ч, запись переживает истечение токена), но не реализована в UI.

**Что сделать.**
- **Экран входа оператора** по JWT-flow `ex_system` (`docs/auth.md`):
  логин/пароль → токены; авто-обновление токена в фоне.
- **Оффлайн-старт по кэшированной сессии**: последний успешный вход хранится
  зашифрованно `auth.operator.cached_session_hours`; в оффлайне старт записи
  разрешён по кэшу с индикацией «оффлайн-режим».
- Идущая запись **никогда** не прерывается истечением токена/выходом
  (`auth.recording_survives_token_expiry`) — блокируется только старт новой.
- Шапка приложения: текущий оператор (ФИО), статус связи с сервером, смена
  оператора между заседаниями без перезапуска приложения.
- `operator_id` из сессии входа — в манифест, журнал событий и выгрузку
  (сейчас — из env); учётка станции (`station_id`) — отдельный транспорт
  выгрузки, как зафиксировано.
- Роль пользователя из `ex_system` — основа разнесения настроек (п. 4).

**Ожидаемый результат.** Юридически значимая привязка записи к оператору без
ручных env-переменных; станция работает и онлайн, и в оффлайн-зале; убрана
последняя «временная» подпорка v1.

## 4. Разнесение настроек администратора и оператора

**Что сейчас.** Один экран «Настройки» с двумя *визуальными* секциями
(«Оператор» и «Администратор») — любой пользователь станции может изменить
что угодно: URL сервера, шифрование, ретеншн, пороги надёжности.

**Что сделать.**
- **Контроль доступа к секции администратора**: по роли из `ex_system`
  (после п. 3) с локальным фолбэком — админ-PIN/парольная фраза станции для
  оффлайна. Для оператора админ-секция видна read-only (прозрачность без
  возможности сломать).
- Разделить экраны: «Настройки» (оператор: устройство, справочники меток/ролей,
  удобства) и «Администрирование» (сервер, шифрование, ретеншн, пороги
  надёжности, многоканальная карта дорожек, разрешение экспорта).
- **Журнал изменений настроек** (кто, когда, что поменял) — в общий журнал
  событий.
- **Экспорт/импорт профиля станции** (JSON без секретов) — тиражирование
  конфигурации на десятки залов и быстрое восстановление станции.
- Валидация «опасных» изменений: смена URL сервера/отключение шифрования — с
  подтверждением и записью в журнал.

**Ожидаемый результат.** Оператор не может случайно нарушить работоспособность
или безопасность станции; администратор разворачивает и сопровождает парк
станций единообразно и с аудитом.

## 5. Дизайн и адаптивность

**Что сейчас.** Макет фиксированный: боковая панель 232 px, контент с
`maxWidth` 760–880 px, стили инлайновые, брейкпоинтов нет. На узком окне,
при масштабировании ОС 125–150 % или на маленьком мониторе в зале интерфейс
не перестраивается. Статус записи — небольшой тег, издалека не читается.

**Что сделать.**
- **Адаптивная сетка**: брейкпоинты для узких окон (сворачивание боковой
  навигации в иконки/бургер), корректная работа при масштабировании ОС и на
  экранах от ~1280×720; проверка тач-ввода (моноблоки в залах).
- **Режим зала («большой статус»)**: разворачиваемая панель с крупным
  хронометром, состоянием записи и уровнем сигнала — читается с нескольких
  метров; опционально компактное окно «поверх всех окон», чтобы оператор
  работал в других программах, не теряя контроль записи.
- **Глобальный индикатор записи** в шапке (виден со всех экранов, а не только
  на «Записи») + иконка в системном трее с состоянием.
- Вынести повторяющиеся инлайн-стили в разделяемые токены/классы поверх PravoUI
  (единообразие, дешевле поддержка); при появлении тёмной темы в дизайн-системе
  — поддержать её токенами.
- Пройтись по доступности: контраст, фокус-обводки, aria (частично уже есть),
  полная работа с клавиатуры.

**Ожидаемый результат.** Приложение выглядит аккуратно на любом мониторе зала,
состояние записи всегда на виду, интерфейс соответствует уровню PravoUI и
привлекателен для пользователей.

## 6. Удобство и привлекательность для пользователей (UX-пакет)

**Что сейчас.** База хорошая (горячие клавиши, подтверждения, баннеры
надёжности, восстановление сессий), но рабочие сценарии секретаря покрыты
не до конца.

**Что сделать.**
- **Проверка перед заседанием** (self-test одной кнопкой): устройство отвечает,
  уровень сигнала в норме, места на диске достаточно, сервер доступен, вход
  выполнен → зелёный чек-лист «можно начинать».
- **Мастер первого запуска** для новой станции: URL сервера, выбор устройства,
  тестовая запись с прослушиванием (после п. 1), проверка выгрузки.
- Экран «Сессии»: **поиск и фильтры** (по № дела, дате, статусу выгрузки),
  сортировка, **карточка сессии** с метками, событиями журнала, статусом
  целостности и действиями «Прослушать»/«Экспортировать»; пагинация на большие
  архивы.
- **Защита от «сюрпризов» ОС** во время записи: блокировка сна/гибернации,
  предупреждение при попытке закрыть приложение с идущей записью, опция
  автозапуска приложения при входе в систему.
- Уведомления о сбоях (обрыв устройства, критический диск, ошибка выгрузки) —
  системные нотификации + опциональный звуковой сигнал (настраивается: в зале
  звук может быть неуместен).
- Мелочи трения: отмена последней метки одной клавишей, комментарий к сессии,
  копирование пути/№ дела из карточки, человекочитаемые тексты ошибок ядра.

**Ожидаемый результат.** Секретарь начинает заседание уверенно (чек-лист),
находит и отдаёт записи без обучения, а станция сама защищает запись от
типовых бытовых сбоев. Продукт воспринимается целостным и дружелюбным.

## 7. Дозавершение серверного приёма (этап 07, в `ex_system`)

**Что сейчас.** Контракт определён и клиент (этап 06) на него опирается, но
серверная реализация неполная.

**Что сделать (по `promts/07_backend_integration.md`).**
- Довести приём до актуального контракта `manifest_version = 3`: мультитрек
  (`tracks[]`, части по `PUT .../part/{track_id}/{part_index}/`) и блок
  `annotations` (метки/роли + хеш-цепочный лог) в `upload/init`.
- Серверная верификация целостности (`verify`: пересчёт SHA-256 и цепочки),
  статусы для поллинга станции, передача в пайплайн W2.11, ретеншн
  (`purge_expired_audio`), троттлинг.
- Синхронно поддерживать пару документов: `07_backend_integration.md` ↔
  `06_sync_agent.md` (единый источник истины контракта).

**Ожидаемый результат.** Статус «Подтверждена» на станции означает реальную
серверную верификацию; метки и роли доезжают до W2.11 и улучшают черновик
протокола; локальный ретеншн безопасно удаляет подтверждённые записи.
