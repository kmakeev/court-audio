# Требования к железу и ресурсоёмкость станции

Оценка ресурсоёмкости «Аудиопротокола» и рекомендации по аппаратной
конфигурации рабочей станции зала. Основана на анализе кода ядра
(2026-07-05, v0.2.0 + этапы 13.x): горячие пути захвата/записи, шифрование,
выгрузка, плеер, экспорт. Числа по умолчанию — из реестра
[`configuration.md`](configuration.md); методика проверки на целевом стенде —
в конце документа.

## Резюме

Приложение **лёгкое**: ресурсоёмкость определяется webview-оболочкой UI
(WebKitGTK на Astra SE / РЕД ОС, WebView2 на Windows), а не Rust-ядром записи.
Ядро спроектировано экономно: lock-free кольцевой буфер без аллокаций в
realtime-пути, сегментная запись через буферизованный ввод-вывод с fsync раз в
1.5 с, посегментная потоковая обработка в плеере и WAV-экспорте, инкрементальное
хеширование, дросселированные события UI (метры 25 Гц, позиция плеера 5 Гц),
блокирующий HTTP-клиент без async-рантайма. Узкие места перечислены в разделе
«Известные ограничения».

## Профиль нагрузки

### Постоянная нагрузка при записи

| Ресурс | 1 канал 44.1 кГц/16 (дефолт v1) | 8 каналов 48 кГц/16 | Примечание |
|---|---|---|---|
| CPU | < 1–2 % одного ядра | < 5 % суммарно | Конвертация f32→i16 + WAV-запись; DSP/ресемпла нет |
| RAM ядра | ~30–60 МБ | + ~1 МБ буферов на дорожку | Кольцо = `capture_buffer_seconds` (2 с) × частота × каналы × 4 Б |
| RAM UI (webview) | 150–400 МБ | так же | Главный потребитель памяти; зависит от ОС/версии движка |
| Диск, поток записи | ~318 МБ/ч | ~2.7 ГБ/ч | WAV PCM: `частота × разрядность/8 × каналы`; ×2 при зеркале |
| Диск, операции | fsync раз в 1.5 с на дорожку | ×8 | На HDD при N дорожек — seek-шторм: нужен SSD |
| Потоки ОС | ~3 на дорожку + база приложения | ~24 + база | Захват + райтер + супервизор на дорожку (изоляция отказов) |

Битрейт одной дорожки: 44.1 кГц/16/моно = 88.2 КБ/с ≈ **318 МБ/ч**;
48 кГц/16/моно ≈ 345 МБ/ч. Многоканал — кратно числу дорожек.

### Пиковые операции (разовые)

| Операция | CPU | RAM | Когда |
|---|---|---|---|
| Вход/PIN (Argon2id) | ~0.1 с одного ядра | ~20 МБ разово | Вход оператора, разблокировка |
| Хеширование SHA-256 | ~1 с на час записи | потоково | Стоп сессии / реконсиляция после рестарта |
| Шифрование сегмента (AES-256-GCM) | пренебрежимо при AES-NI | ~2× сегмента (≈5 МБ) | Закрытие сегмента (после этапа 13.7) |
| Выгрузка в `ex_system` | TLS, доли ядра | ~2× части (≤16 МБ) на поток | Фон; `parallel_uploads` = 1 |
| Экспорт WAV | доли ядра | один сегмент (~2.6 МБ) | Потоковая склейка |
| **Экспорт FLAC** | 1 ядро, минуты | **⚠ вся сессия в памяти** (см. ограничения) | До исправления R-014 |

## Критерии использования

### Максимум параллельно записываемых каналов

CPU практически не ограничивает (и 16 дорожек — проценты ядра). Реальные
ограничители, по значимости:

1. **Аудиоподсистема ОС.** Каждая дорожка открывает собственный входной стрим
   устройства (осознанная архитектура «сессия на дорожку»: обрыв одной дорожки
   не роняет остальные). Несколько стримов на одно устройство требуют
   WASAPI shared mode (Windows — работает из коробки) или dsnoop/PulseAudio
   (Linux — проверить на целевой Astra SE / РЕД ОС).
2. **Диск.** N дорожек = N параллельных файлов с fsync каждые 1.5 с. На SSD
   запас многократный; HDD допустим только для 1 канала без зеркала.
3. **Практическая рекомендация: до 6–8 дорожек** (совпадает со справочником
   ролей по умолчанию). Связку «аудиоинтерфейс + драйвер + N дорожек»
   обязательно проверить на целевом стенде до ввода зала в эксплуатацию.

### Параллельные выгрузки по сети

Дефолт `sync.parallel_uploads = 1` — рекомендуемый: узкое место — сервер
`ex_system` и сеть зала, не станция. Каждый поток выгрузки держит в памяти
~2× размер части (часть = целый сегмент, ≤ `sync.chunk_size_mb`), сервер
читает часть целиком в память — поднимать выше **2** не стоит без явной
необходимости. Ориентир скорости догрузки: часовая моно-сессия (~320 МБ) по
100 Мбит — ~1 минута чистой передачи; очередь и ретраи с бэкоффом работают в
фоне и записи не мешают.

### Диск: формула ёмкости

```
V ≈ 0.32 ГБ/ч × дорожки × часы_записи_в_день × дни_локального_хранения
    × (1 + зеркало) × 1.2 (запас)
```

Дни локального хранения определяет ретеншн: по умолчанию копия удаляется после
подтверждённой выгрузки + 72 ч (`retention.*`); в **изолированном зале**
записи копятся до выгрузки/экспорта — считать на всё окно автономной работы.
Пример: 2 дорожки × 6 ч/день × 5 дней × без зеркала ≈ **23 ГБ**; изолированный
зал на квартал (1 канал, 4 ч/день, 60 рабочих дней) ≈ **92 ГБ**.

Пороги контроля места (`reliability.disk_low_threshold_mb` = 1 ГиБ,
`disk_critical_mb` = 256 МиБ) — минимальны; на многоканальных станциях поднять
кратно битрейту (например, low = 2 часа записи).

## Рекомендуемые конфигурации

| | Минимум (1 канал) | Рекомендуется (многоканал / зеркало / экспорт) |
|---|---|---|
| CPU | x86_64, 2 ядра ≥ 2 ГГц, **с AES-NI** | 4 ядра (запас на FLAC-экспорт и webview) |
| RAM | 4 ГБ (с оговоркой FLAC — см. ограничения) | **8 ГБ** |
| Диск | SSD SATA 128–256 ГБ | SSD 480+ ГБ + отдельный носитель под зеркало |
| Сеть | Ethernet 100 Мбит | 1 Гбит |
| Аудиовход | Встроенный/USB-микрофон | USB-аудиоинтерфейс по числу дорожек |
| Питание | **ИБП обязателен** | ИБП + настроенный запрет сна ([`os_integration.md`](os_integration.md)) |

Примечания по ОС:

- **Astra SE / РЕД ОС**: память UI — WebKitGTK 4.1 (обычно 200–400 МБ);
  зависимостей за пределами таблицы [`packaging.md`](packaging.md) нет.
- **Windows 10/11**: WebView2 Runtime (~150 МБ на диске в фикс-версии для
  оффлайн-установки); WASAPI shared mode для многоканала.
- ИБП — часть контура «бесперебойности»: софт гарантирует потерю ≤ интервала
  fsync (1.5 с) при обрыве питания, но корректное завершение сессии и
  сохранность файловой системы обеспечивает питание.

## Известные ограничения (до этапа 13.7)

Найдены анализом кода, приняты в работу —
[`../promts/13_7_flac_memory_and_segment_encryption.md`](../promts/13_7_flac_memory_and_segment_encryption.md):

1. **Экспорт в FLAC держит всю сессию в памяти** (R-014): ≈ 1.9 ГБ на 3 ч
   моно 44.1 кГц, кратно больше для многоканала. До исправления: на станциях
   с 4 ГБ RAM экспортировать длинные сессии в **WAV** (потоковый путь) либо
   разбивать; FLAC — для коротких записей.
2. **Аудио-сегменты на диске пока не шифруются** (R-013): ключом станции
   шифруются служебные данные (кэш сессии, PIN-хеши, кэш дел), а сегменты
   лежат открытым WAV — учитывать при физической защите носителя станции
   (диск с ОС-шифрованием тома компенсирует до подключения, см. промт 13.7).

## Методика проверки на целевом стенде

При вводе зала (особенно многоканального) в эксплуатацию:

- [ ] Запись N дорожек ≥ 1 ч: в «Диагностике» нет xrun/дропов (счётчик
      кольцевого буфера), сегменты ровные, хеш-цепочка сходится.
- [ ] Мониторинг ресурсов во время записи (`top`/Диспетчер задач): CPU ядра
      < 10 %, RSS процессов стабилен (нет роста от часа к часу).
- [ ] Параллельно записи — выгрузка накопленной очереди: запись не деградирует
      (метры живые, сегменты без пропусков).
- [ ] Экспорт типовой сессии зала в выбранном формате: время и пиковая память
      приемлемы (до 13.7 — только WAV для длинных сессий).
- [ ] Обрыв питания под записью (стенд!) — после старта сессия восстановлена,
      потеря ≤ 2 с хвоста.
