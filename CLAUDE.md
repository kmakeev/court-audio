# CLAUDE.md

Руководство для Claude Code при работе с проектом **«Аудиопротокол»** —
внешним модулем аудиопротоколирования судебных заседаний.

## Описание проекта

«Аудиопротокол» — самостоятельное кроссплатформенное ПО для **надёжного
ведения аудиозаписей** в залах судебных заседаний и кабинетах. Запускается на
рабочей станции в зале, обеспечивает бесперебойную и качественную запись,
привязывает запись к делу и выгружает её вместе с метаданными в основную
экспертную систему (`ex_system`), где запись подхватывает обрабатывающий модуль
**W2.11 «Аудио-модуль помощника»** (ASR/диаризация/NER/протокол).

Функциональный ориентир — **SRS «Фемида»** (аудио-видеопротоколирование
заседаний с метками, защитой целостности, архивом). Отличие: в v1 — **только
аудио**, видео и юридически значимая ЭЦП — фаза 2.

### Граница ответственности (важно)

| Часть | Где живёт | Что делает |
|---|---|---|
| **Захват** (этот проект) | `~/Progs/court-audio/` | Пишет звук, обеспечивает бесперебойность, привязывает к делу, выгружает в `ex_system` |
| **Приём + обработка** | `~/Progs/ex_system/` (Django + W2.11) | Принимает запись (этап `07`), ASR/диаризация/NER, черновик протокола |

Точка стыковки — модель `AudioRecording`, заложенная в контракте W2.11

обработку оставляем в `ex_system`. **Не дублировать ASR/NER в этом проекте.**

## Технологический стек

- **Ядро захвата:** Rust (`src-tauri/`). Звук — `cpal`; кодирование — WAV (PCM)
  без потерь как мастер + опц. FLAC. Локальный манифест — SQLite (`rusqlite`).
- **Оболочка:** Tauri 2.x (системный webview, лёгкий бинарь под все целевые ОС).
- **UI:** React 19 + TS + Vite, дизайн-система **«Правовой кодекс» (PravoUI)** —
  переиспользуется из `ex_system_front` через `.design-sync` (токены/шрифты/
  компоненты; вручную CSS-переменные не задавать).
- **Импортозамещение:** сборка и работа на отечественных Linux; без зарубежных
  облачных зависимостей; перспектива внесения в Реестр отечественного ПО.

## Принципы «бесперебойности» (нефункциональное ядро)

Запись не должна прерываться и не должна терять более нескольких секунд при
любом сбое. Обеспечивается слоями `recorder/` + `reliability/`:

1. **Сегментная запись с непрерывным сбросом** — данные пишутся на диск
   короткими сегментами (напр. 10–30 с); потеря питания теряет максимум
   незавершённый сегмент.
2. **Журнал-манифест (write-ahead)** — состояние сессии и список сегментов
   фиксируются до/во время записи; при рестарте сессия восстанавливается.
3. **Watchdog + авто-рестарт** процесса записи.
4. **Обработка отказов устройства** — отключение микрофона не роняет сессию
   (пауза + индикация + авто-возобновление при возврате устройства).
5. **Контроль ресурсов** — мониторинг свободного места и здоровья устройства.
6. **Опц. дублирующая дорожка** на второй носитель (конфиг).

## ПДн и безопасность

Аудио заседаний — чувствительные данные. Требования:

- Шифрование записей at-rest на станции (`store/`).
- Политика ретеншна локальных копий (по умолчанию — удалять после успешной
  выгрузки + подтверждения; настраиваемое окно). Зеркало подхода `ex_system`
  (`purge_expired_uploads` + Celery beat, раунд 06.ops).
- Контроль целостности (SHA-256 + хеш-цепочка) и журнал событий записи.
- Аутентификация: **оператор обязан авторизоваться** (вход — для старта
  сессии; идущая запись не прерывается истечением токена; оффлайн-старт — по
  кэшированной сессии). У станции — своя учётка на зал (транспорт выгрузки).
  Фиксируем `station_id` + обязательный `operator_id`. JWT текущей системы
  (`docs/auth.md` в `ex_system`).
- В макетах/демо — **только синтетика, без реальных ПДн**.
